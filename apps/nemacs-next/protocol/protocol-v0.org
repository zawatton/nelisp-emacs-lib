#+TITLE: nemacs-next protocol V0
#+AUTHOR: nelisp-emacs
#+DATE: 2026-06-25
#+STATUS: DRAFT

* Goal

Protocol V0 is a persistent message protocol between a modern GUI frontend
and an editor session that owns Emacs-observable state.

The first transport can be newline-delimited JSON for inspection.  The
semantic contract should not depend on JSON specifically.

The current app scaffold provides a minimal encoder in
=apps/nemacs-next/lisp/nemacs-next-protocol.el=.  It exists only to prove
the persistent session can return =hello= and =snapshot= messages; a richer
transport can replace it without moving editor semantics into the GUI.

* Client Messages

| Type | Required fields | Meaning |
|------+-----------------+---------|
| =hello= | =protocol=, =capabilities= | negotiate version and frontend capabilities |
| =input= | =event= | keyboard, mouse, or IME event after raw frontend decoding |
| =command= | =name=, =args= | command request from menu, toolbar, palette, or native action |
| =resize= | =frame=, =width=, =height= | viewport or frame dimensions changed |
| =open= | =path= | OS file-open request |
| =clipboard= | =request-id=, =payload= | clipboard response or pushed clipboard data |
| =shutdown= | =reason= | graceful shutdown request |

* Session Messages

| Type | Required fields | Meaning |
|------+-----------------+---------|
| =hello= | =protocol=, =capabilities= | accepted session capabilities |
| =snapshot= | =frame=, =buffers= | complete state needed to paint a frame |
| =delta= | =frame=, =changes= | incremental state update |
| =minibuffer= | =active=, =prompt=, =contents= | minibuffer state and optional candidates |
| =echo= | =message= | echo-area update |
| =menu= | =items= | menu, toolbar, or tab model |
| =request= | =request-id=, =op= | frontend OS integration request |
| =error= | =code=, =message= | structured command/session error |

* M1 Command Subset

The current proof supports a deliberately tiny command subset:

| Name | Meaning |
|------+---------|
| =create-buffer= | allocate and select a buffer through =nelisp-ec-generate-new-buffer= |
| =insert-text= | insert a string through reusable buffer APIs |
| =snapshot= | return the current buffer snapshot; kept out of process smoke until repeated post-edit snapshot is stable in NeLisp REPL |

This subset exists to prove the persistent process boundary.  Full keymap
lookup and command-loop dispatch still belong in =nelisp-emacs= core.

* M2 Movement and Delete Subset

The next slice of the M2 editing loop adds point movement and character
deletion through the same reusable buffer APIs used by =insert-text=:

| Name | Meaning |
|------+---------|
| =forward-char= | move point forward by optional :count (default 1) through =nelisp-ec-forward-char= |
| =backward-char= | move point backward by optional :count (default 1) through a negated =nelisp-ec-forward-char= delta |
| =goto-char= | move point to an absolute :position through =nelisp-ec-goto-char= |
| =delete-char= | delete :count characters (default 1; negative = backward) through =nelisp-ec-delete-char= |

Each of these commands returns a buffer snapshot on success, same as
=insert-text=.  Movement or deletion that would leave the buffer (or
narrowed region) bounds returns a structured =out-of-range= protocol
=error= payload instead of raising an uncaught signal in the persistent
session -- an editing loop must be able to reject an out-of-bounds request
and keep running.

* M2 Newline, Undo, Kill, and Yank Subset

The remaining M2 editing-loop slice adds newline handling, single-step
undo, and a kill-ring round trip, closing the milestone:

| Name | Meaning |
|------+---------|
| =newline= | insert optional :count newline characters (default 1) at point |
| =undo= | undo one edit group on the current buffer |
| =kill-region= | kill the text between required :start and :end through the kill ring |
| =kill-line= | kill from point to end of line (or the trailing newline at EOL) through the kill ring |
| =yank= | insert the current kill-ring head at point |

Implementation notes:

- =newline= inserts through =nelisp-ec-insert=, the same reusable
  primitive =insert-text= uses -- not through the unprefixed
  Emacs-compatible =newline= command.  A literal =\n= in =insert-text='s
  =:text= already takes this same path and was already valid before this
  slice; =newline= exists as a named convenience for frontend Enter-key
  handling.  This distinction matters because =emacs-edit-builtins= only
  installs its unprefixed polyfills (=newline=, =kill-region=,
  =kill-line=, =yank=, =kill-new=) when host Emacs does not already
  provide that name; under the host-Emacs smoke path
  (=scripts/smoke.sh=) those unprefixed names resolve to the *host's*
  own buffer, not the =nelisp-ec= session buffer, so the session layer
  only calls the =nelisp-ec-*= primitives and the prefixed
  =emacs-edit-*-direct= / =emacs-undo-*= APIs, never the bare
  Emacs-compatible command names.
- =undo= is a real substrate feature, not a session-layer fake: it
  delegates to =emacs-undo-undo-direct= (Doc 51 Track E.2, already
  shipped in =packages/nelisp-emacs-editing/lisp/emacs-undo.el= /
  =src/emacs-undo.el=), which tracks a per-buffer undo list independent
  of host Emacs's native =buffer-undo-list=.  Because =insert-text= and
  =delete-char= mutate the buffer through the raw =nelisp-ec-insert= /
  =nelisp-ec-delete-char= primitives (bypassing the automatic recording
  built into unprefixed commands like =self-insert-command=), the
  session layer explicitly calls =emacs-undo-record-insert= /
  =emacs-undo-record-delete= around those two commands so their edits
  are not silently invisible to undo.  Every buffer-mutating command
  (=insert-text=, =delete-char=, =newline=, =kill-region=, =kill-line=,
  =yank=) closes its own undo group with =emacs-undo-undo-boundary=
  after it completes, so one =undo= command reverts exactly the edit
  made by the single most recent mutating command.
- =kill-region=, =kill-line=, and =yank= delegate to
  =emacs-edit-kill-region-direct=, =emacs-edit-kill-line-direct=, and
  =emacs-edit-yank-direct= (Track E, already shipped in
  =emacs-edit-builtins.el=), which read/write the shared =kill-ring=
  list.  =yank= with an empty kill ring does not silently no-op insert
  nothing; it returns a structured =empty-kill-ring= protocol error.
  =kill-region= without integer =:start=/=:end= returns a structured
  =bad-command= error; out-of-buffer-bounds =:start=/=:end= returns
  =out-of-range=, matching the existing =goto-char=/=delete-char= error
  shape.
- =undo= with no further undo information (a fresh buffer, or a buffer
  whose undo history has already been fully consumed) returns a
  structured =no-further-undo-information= protocol error instead of an
  uncaught signal; the session stays alive and can keep handling
  commands afterward, same as the =out-of-range= movement/delete errors.

* M3 Minibuffer/File Loop

M3 adds the file and buffer loop named in Doc 31 Section 7:

| Name | Meaning |
|------+---------|
| =find-file= | visit required =:path= through reusable file I/O APIs and return a snapshot |
| =save-buffer= | save the current visited buffer through reusable file I/O APIs and return a snapshot with =:saved-file= |
| =switch-to-buffer= | select required =:buffer-name= through reusable buffer APIs and return a snapshot |
| =kill-buffer= | kill =:buffer-name= or the current buffer through reusable buffer APIs and return the next snapshot |
| =complete= / =completion= | return a =minibuffer= payload with candidate strings for =:purpose= / =:input= |

The protocol-level =open= message is treated as an OS file-open request
and delegates to the same =find-file= path.

Implementation notes:

- The app session calls =emacs-fileio-visit-file-direct= and
  =emacs-fileio-save-buffer-direct= when available, falling back only to
  reusable Emacs-compatible file commands if the direct helpers are not
  loaded.  GUI/frontend code does not read files, write files, or decide
  visited-file state.
- =switch-to-buffer= and =kill-buffer= use the reusable buffer registry
  and =nelisp-ec-set-buffer= / =nelisp-ec-kill-buffer= path.  A missing
  kill target returns a structured =no-such-buffer= protocol =error= and
  the session stays alive.
- Completion responses have =:type minibuffer=, =:active t=,
  =:purpose=, =:prompt=, =:contents=, =:completion=, and =:candidates=.
  Buffer completion derives candidates from the reusable buffer list.
  File completion derives candidates from the reusable directory/file-name
  APIs when available.  Tests also cover an explicit =:collection= list
  so frontend completion refresh can be validated without file-system
  dependence.
- Successful file and buffer commands return full buffer snapshots.  The
  snapshot now includes =:file-name= and =:modified= so a frontend can
  render visited-file and dirty-state indicators without owning the
  semantics.

* Rule

Protocol messages may carry editor state, but they must not move editor
semantics into the GUI.  If an Emacs package can observe a state change,
the persistent session and reusable libraries own it.
