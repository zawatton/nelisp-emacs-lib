This memo classifies the missing rows from the GUI keymap coverage script.

The classification is a primary-blocker classification.  Some commands cross
boundaries, for example =project-shell-command= is both project-facing and
process-backed.  In those cases the bucket below records the next blocking
dependency for a usable GUI replacement.

No runtime implementation file was changed for this memo.

| Priority | Item | Bucket | Reason | Touch layers |
|----------+------+--------+--------+--------------|
| 1 | Window teardown and placement: =quit-window=, =delete-windows-on=, =toggle-window-dedicated=, =window-configuration-to-register= | 5 | Many help, dired, project, vc, and temporary-buffer flows depend on predictable window cleanup before their package-specific behavior matters. | =nelisp-emacs= first; =nelisp-gui= repaint only |
| 2 | Window sizing: =enlarge-window=, =enlarge-window-horizontally=, =shrink-window-horizontally=, =fit-window-to-buffer= | 5 | High-frequency GUI ergonomics, contained in the existing window model, and useful before adding more package buffers. | =nelisp-emacs=; =nelisp-gui= if snapshot lacks size fields |
| 3 | Basic frame commands: =make-frame-command=, =delete-frame=, =other-frame=, =find-file-other-frame=, =switch-to-buffer-other-frame= | 5 | A GUI replacement eventually needs multi-frame behavior; the semantic model must exist before native X11 frame lifecycle can be trusted. | =nelisp-emacs= model plus =nelisp-gui= frame creation/destruction |
| 4 | Basic tab commands: =find-file-other-tab=, =switch-to-buffer-other-tab=, =tab-switch=, =tab-new-to=, =tab-close-other= | 5 | Tab-bar workflows are keymap-visible and mostly pure editor state once buffers/windows are stable. | =nelisp-emacs=; =nelisp-gui= only for tab display |
| 5 | Input-method entry points: =toggle-input-method=, =set-input-method=, =activate-transient-input-method= | 6 | This is a daily-driver blocker for Japanese and other non-ASCII input. | 6 | Coding-system command prefix: =universal-coding-system-argument= and =C-x RET *= setters | 6 | File/process/selection coding needs explicit user control before the GUI can replace Emacs on mixed-encoding projects. | =nelisp-emacs= wrappers over =nelisp= conversion/coding support |
| 7 | Project navigation: =project-find-file=, =project-switch-project=, =project-dired=, =project-switch-to-buffer= | 4 | Large daily coding payoff without requiring full VC parity; can be implemented as a narrow project facade. | =nelisp-emacs=, with filesystem primitives from =nelisp= if needed |
| 8 | Project process commands: =project-compile=, =project-shell-command=, =project-async-shell-command= | 3 | The process/syscall work should be exercised by real editor workflows; these commands expose async process behavior in the GUI. | =nelisp= process/syscall plus =nelisp-emacs= process/project semantics |
| 9 | Error navigation: =next-error=, =previous-error= | 4 | Completes the compile/search feedback loop after process-backed commands produce buffers. | =nelisp-emacs= |
| 10 | VC essentials: =vc-dir=, =vc-diff=, =vc-next-action=, =vc-revert=, =vc-root-diff= | 4 | Repository workflows are important, but should follow project/process plumbing so the implementation can stay narrow and process-backed. | =nelisp-emacs= over =nelisp= process/syscall |

Cheap coverage wins not in the top ten: =kmacro-*=, =expand-abbrev=,
=add-*-abbrev=, and =rectangle-mark-mode= are mostly self-contained in
=nelisp-emacs=.  They are good filler tasks when the larger frame/tab/project
work is blocked, but they are not the main replacement blockers in this
coverage run.
