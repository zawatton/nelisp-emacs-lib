* 0. Purpose

Task #16.  Makes =/ssh:HOST:/path= usable in nelisp-emacs: vendored
Emacs Tramp (=vendor/emacs-lisp/net/tramp*.el=, version 2.7.1.30.1) is
brought up on the ssh/scp methods only, with the file-name-handler
dispatch substrate Tramp needs and every other Tramp method blocked with
an explicit signal instead of an obscure half-loaded failure.

Approved scope (2026-06-10 user decision, =docs/design/11-remaining-work-roadmap.org=
M11): ssh/scp file read/write/process round-trip.  Dired full
integration, multi-hop (=/ssh:a|ssh:b:/path=), interactive pty/password
auth, and every non-ssh/scp Tramp method are explicitly out of scope.

Naming note: =vendor/nelisp/packages/nelisp-tramp/= is the NeLisp
trampoline evaluator (public API =nelisp-tramp-eval=) and is unrelated to
this work; nothing in this lane is named =nelisp-tramp-*=.

* 2. Substrate gaps found and closed

- *file-name-handler-alist had no dispatcher.* =find-file-name-handler=
  did not exist as a standalone primitive; the 16 file-I/O primitives in
  =emacs-fileio-builtins.el= were bare =nelisp-ec-*= aliases with no
  handler check at all.  Closed by =emacs-file-name-handler.el= +
  the =emacs-fileio-builtins.el= dolist split (Sec 1).  All host-inert.
- *=process-mark= was hard-wired nil* in the standalone bidi process
  layer.  =tramp-sh=/=comint= read =process-mark= to track how much
  output they have consumed.  Closed with a lazily created real marker
  in bidi-process vector slot 13, kept in sync by
  =emacs-process--bidi-drain='s insert path.
- *=start-file-process=/=process-file= were absent* from the bare
  standalone preload script (only present at the =src/emacs-process.el=
  layer, which a bare preload image does not load).  Added at both
  layers for defence in depth.
- *=accept-process-output='s native (non-bidi) path ignored SECONDS*,
  doing a single non-blocking poll.  Given a 10ms-slice retry loop up to
  the requested budget, mirroring the bidi layer's existing pattern
  (insurance -- Tramp's synchronous connection setup can block on this
  call for multi-second stretches).

* 3. The hermetic stub-ssh discovery (read before touching the test)

=test/nemacs-tramp-stub-ssh-test.el= installs a fake =ssh= executable
ahead of =exec-path=/PATH that ignores every ssh(1) argument and execs a
real local shell in its place.  This works because the default "ssh"
Tramp method opens one persistent shell session over ssh's stdin/stdout
with no trailing command (see =tramp-methods= in =tramp-sh.el=), so Tramp
does not depend on the real ssh(1) wire protocol once a shell answers.

The first implementation (=exec /bin/sh=) hung indefinitely.  Diagnosed
with =strace -f=: the spawned shell sat parked on a blocking =read(0,
...)= forever, because Tramp's own first write to the channel never
happened.  =tramp-actions-before-shell= makes
=tramp-open-connection-setup-interactive-shell=/=tramp-process-actions=
wait to observe a shell prompt (=shell-prompt-pattern=/
=tramp-shell-prompt-pattern=) before sending anything at all -- and a
POSIX shell only emits its PS1 prompt when it considers itself
*interactive*, and conventionally writes that prompt to *stderr*.  The
fix is =exec /bin/sh -i 2>&1=: `-i` turns on the prompt, `2>&1` puts it
on the channel Tramp actually reads (stdout).  Without both flags the
wait has no bound reachable in this configuration and the test hangs
forever rather than failing fast.

This is the single reusable fact from this whole investigation: *any*
future hermetic Tramp/ssh test-double in this repository needs
=-i 2>&1= (or equivalent) on its fake login shell.

* 5. Known limitations / out of scope

- Only "ssh" and "scp" methods are accepted; every other method
  (=smb=, =adb=, =sudo=, =su=, =rsh=, =telnet=, =rclone=, =fuse=,
  =crypt=, =container=, =archive=, =ftp=, ...) signals
  =nemacs-tramp-unsupported-method= via the =tramp-dissect-file-name=
  guard.
- No multi-hop (=/ssh:a|ssh:b:/path=).
- No pty/interactive password prompting: =tramp-process-connection-type=
  is forced nil (pipe); key-based auth only.  =auth-sources= is nil so
  Tramp never touches =~/.authinfo=/=~/.netrc=.
- No Dired integration beyond whatever =tramp-sh='s own generic
  =directory-files=/=insert-directory= handler already provides; a
  polished Dired UX is explicitly a separate task.
- =rename-file='s handler dispatch only inspects FILE, not NEWNAME
  (cross-host rename is out of scope for the ssh-only lane).
- The standalone-reader (NeLisp, not host Emacs) full Tramp load chain
  is unverified (Sec 4).
