* 2. Breadth loop playbook (再現手順)

1. *package 選定*: foundational 優先 (=dash= / =s= / =ht= / =cl-lib=)。依存先が
   広いほど 1 fix の波及が大きい。
2. *駆動*: full --batch =--eval= で =(load PKG)= → 関数を exercise。各呼び出しを
   =condition-case= で包み void-function を捕捉 (§4 の安定性に注意)。
3. *分類*: 失敗を「*単純 void-function* (逐次追加)」と「*深い runtime 機構*
   (dedicated)」に切り分け (§3)。
4. *補完*: 単純は polyfill。=(unless (fboundp 'X) (defun X ...))=。host の C builtin
   を守る必要があれば =(when (fboundp 'rdf) ...)= で reader-gate (= getenv /
   replace-match と同パターン)。
5. *bundle*: =make build-nelisp-bootstrap= 再生成 → reader で検証
   (=./bin/nemacs --driver=nelisp --batch --eval ...=)。
6. *test + commit*: host test (source-shape or 純関数) + 必要なら reader-gate test。
   .elc を canonical に残さない。

* 3. 単純 vs 深い の判定基準

- *単純* = 1〜数行 defun で済む。例: =cl-evenp= =(= 0 (% x 2))=、=function-get=
  (=get= + alias 追従)、=replace-match= string 形 (=match-beginning/end= + concat)。
- *深い* = runtime 機構と絡む。例: =define-inline= (runtime backquote が =,X= を
  =(comma X)= で read する表現差)、=macroexpand-all=、buffer mutation 系
  (=replace-match= buffer 形)。
- *既解決の深い例*: =define-inline= → inline.el 機構を捨て、=(comma X)=→X 置換の
  lean polyfill を実装 (Doc 15 §3.5)。深いものも「runtime の実装を理解すれば
  汎用 polyfill に落ちる」ことがある。

* 4. ⚠️ reader-load 安定性 gotchas (testing 前に必読)

- *stale lock で boot hang*: =/tmp/nemacs-transport.lock= (dir) + leftover
  =target/nelisp= proc が残ると次 boot が hang/timeout する。*test 前に必ず*
  =rm -rf /tmp/nemacs-transport.lock; pkill -9 -f target/nelisp=。
  - ⚠️ *2026-06-21 訂正*: =pkill -9 -f target/nelisp= は危険。env var の binary
    path が自 shell の argv に self-match して shell ごと SIGKILL され (出力欠落で
    boot hang と誤認しやすい)、並行セッション (dev/nelix perf 等) の reader も
    巻き添えにする。本 round では掃除無しでも boot は 16s で成功。lock が本当に
    stale な時のみ lock dir 単独 =rm -rf= に留め、proc kill は PID 明示で行う。
  - *静的 gap 分析の併用*: full package load を起動せずとも「bare reader の
    =fboundp= probe (=target/nelisp --load= 小ファイル) × bundle =src/*.el= grep」
    で void builtin を発見できる。reader 起動回数を最小化でき host 不安定/並行
    contention を避けられる (Doc 15 §3.5 round 1 で実証)。
- *=-l FILE= で大ファイル nested load → segfault*。=--eval= 直書きを使う。
- *大 form の =--eval= も repl-boot で出力欠落* することがある。小分け + =timeout=。
- 大 package load は CPU/memory 重。host は MCE/memory 既知不安定
  ([[project_debian_host_mce_umc_crash]])。
- これらは *reader-core / host 側 = nelisp-emacs 範囲外* (vendor/nelisp 改変は別 repo
  の制約下)。breadth は =--eval= 小分け + clean lock で回避する。

* 6. 検証パターン (本セッションで確立)

- *host source-shape pin*: reader-only 機能を host で「source に実装が在る」ことで
  固定 (reader を起動せず regression 防止)。例: replace-match gate の
  =(fboundp 'rdf)=、define-inline の "not implemented" 消失。
- *helper を gate 外・無条件定義*: 作用のない pure helper は無条件に defun し host
  で unit-test、macro / 副作用のみ =rdf= gate (define-inline の lowering helper)。
- *reader-gate test*: =skip-unless reader= + =exec/eval-runtime-image=。bridge は
  stdout 非捕捉なので =nl-write-file= で結果を file に書き host で読み戻す。
- bundle 再生成必須 (full --batch は =build/nemacs-bootstrap.el= = gitignore
  artifact を使う)。canonical に =.elc= を残さない。

* 7. tracking

本 loop は *open-ended*。1 round = 1 package or 数 builtin。round ごとに Doc 15
§3.5 に追記し、commit する。「単純は逐次、深いは dedicated」の規律を保てば、
実 package が広く callable になる方向へ確実に前進する。
