* 0. Purpose

This document audits the non-stable rows that remain after the package API
promotion pass.

The goal is to distinguish real unfinished reusable API work from
intentional compatibility, loader, and internal implementation surface.
After the =core/command-loop= promotion, the generated promotion queue has
no =promote-ready=, =needs-doc=, =needs-test=, =needs-review=,
=compat-shim-review=, =compat-shim-helper-review=, or
=validation-helper-review= rows.

This audit is not a new stability contract.  The authoritative stable
package APIs remain the facade query APIs documented in Doc 25:

#+begin_src emacs-lisp
(nelisp-emacs-library-stable-api-symbols)
(nelisp-emacs-library-stable-lazy-api-symbols)
#+end_src

* 2. Decisions

** 2.1 Compatibility globals

=compat-shim-covered= rows are intentional Emacs-compatible names.  They
may be useful when a consumer wants a compatibility layer, but they are not
the preferred new integration surface.  New consumers should use prefixed
stable APIs from the owning package whenever possible.

The current compat-global rows include =dired-mode-map= and
=inferior-emacs-lisp-mode-map=.  These are the canonical keymaps for the
legacy =dired= and =ielm= entry points.  They remain compatibility globals
so existing callers can keep using the established command names, but new
reusable integrations should treat them as compatibility-only and prefer
the prefixed package APIs from Doc 25.

The text-core compat-global rows =rx-constituents=, =rx-to-string=,
=rx-let-eval=, =rx-let=, and =rx-define= are legacy rx compatibility
entry points.  They preserve the historical rx extension surface for code
that still defines or expands the old forms, but new reusable consumers
should prefer the namespaced regex APIs from =nelisp-regex= and treat the
legacy rx forms as compatibility-only syntax support.

These rows stay outside the stable package API unless the project decides
that a specific global name is safe and valuable as a versioned
compatibility promise.  That decision must include:

- explicit host-Emacs override policy;
- focused tests for the shim installation path;
- consumer-facing documentation that says the name is a compatibility
  global, not the primary prefixed API.

** 2.2 Builtins shim helpers

=compat-shim-helper-covered= rows are prefixed helpers defined inside
=emacs-*-builtins.el= modules.  Their prefix makes them visible in the
package API inventory, but their ownership is compatibility support.
External reusable consumers should not depend on these names directly
unless a later review promotes a non-shim equivalent or deliberately
promotes that exact helper into the stable facade manifest.

Preferred future treatment:

- If a helper has broad non-shim reuse value, move or wrap the reusable
  behavior in the owning library module and promote that non-shim API.
- If a helper only exists to install or back an unprefixed Emacs builtin,
  keep it as =compat-shim-helper-covered=.
- If a helper is only an implementation detail, add it to the reviewed
  internal list in the promotion queue generator and keep it out of
  consumer docs.

** 2.3 Reviewed internal helpers

=compat-shim-helper-internal= and =validation-helper-internal= rows were
reviewed and intentionally left as implementation surface.  They should
not appear in external examples or consumer package docs.  If a future
consumer needs one of these behaviors, add a new stable prefixed API at the
owning package boundary instead of exposing the helper directly.

** 2.4 Loader manifests

=loader-manifest= rows are the =*-features= variables that define group
loader membership.  They are important for package extraction and smoke
tests, but they are not stable package API symbols.  Consumers should use
the facade query APIs:

#+begin_src emacs-lisp
(nelisp-emacs-library-package-features 'core)
(nelisp-emacs-library-package-lazy-features 'core)
#+end_src

The loader variables remain stable indirectly through the package loader
contract and =make nemacs-library-package-smoke=.

* 3. Package Notes

** 3.1 =nelisp-emacs-foundation=

Residual rows are pure compatibility globals plus the loader manifest:
=cl-lib-version=, =file-name-sans-extension=, =hash-table-empty-p=,
=hash-table-keys=, =hash-table-values=, =null=,
=process-environment=, and =emacs-foundation-features=.

These names emulate host/global Emacs behavior or publish loader
membership.  The stable reusable surface is already represented by the
prefixed foundation APIs in Doc 25.

** 3.2 =nelisp-emacs-text-core=

The only residual row is =emacs-text-core-features=.  Text-core has no
compatibility globals or shim helper residuals after the current promotion
pass.

The legacy rx compatibility forms are listed in the generated compat-global
inventory rather than the stable text-core API.  They are kept for source
compatibility with older rx syntax, not as the preferred reusable regex
surface.

** 3.3 =nelisp-emacs-buffer-core=

Buffer-core residuals are one compatibility variable, builtins shim
helpers for host-like buffer primitives, two reviewed internal helpers,
one loader manifest, and one validation helper.

Do not promote the =emacs-buffer-builtins-*= helper names directly just
because they have test evidence.  The stable consumer path is the
=nelisp-ec-*=, =emacs-buffer-*=, and =text-buffer-*= API already listed in
Doc 25.  =nelisp-ec-check-live= remains internal validation surface.

The =dired-mode-map= global is kept as the compatibility keymap for the
legacy =dired= entry point.  It is useful for code that already knows the
classic keymap name, but new reusable consumers should prefer the
prefixed =emacs-dired-min= helpers and treat the raw map as compatibility
surface only.

** 3.4 =nelisp-emacs-editing=

Editing residuals are all builtins shim helpers.  Some of them calculate
useful positions or ranges, but the stable external API should stay at the
direct command/result layer from Doc 23 and Doc 25 unless a consumer needs
a smaller reusable primitive.

For future promotion, prefer a new non-shim API or an existing
=emacs-edit-*-direct= function over exposing helper calculators from
=emacs-edit-builtins.el=.

** 3.5 =nelisp-emacs-io=

IO residuals are one compatibility runtime predicate, fileio builtins
helpers, internal direct file helpers, and the loader manifest.

The stable IO package contract remains the prefixed process, standalone,
and =nelisp-ec-*= file/path APIs from Doc 25.  File command wrappers such
as find/save/write-file remain shim support unless an external package
needs a frontend-neutral command result API.

** 3.6 =nelisp-emacs-core=

Core residuals are mostly compatibility globals for window/help/info and
two minibuffer variables.  The only prefixed helper residual is
=emacs-command-loop-kill-emacs=, which backs the =kill-emacs= shim path and
should not become a stable library API without a clear application
shutdown contract.

Use the stable =emacs-window-*=, =emacs-help-*=, =emacs-info-*=,
=emacs-minibuffer-*=, and =emacs-command-loop-*= APIs for new reusable
integrations.

The =inferior-emacs-lisp-mode-map= global is intentionally retained as the
compatibility keymap behind =ielm=.  External consumers that want the
editor-independent REPL behavior should depend on the prefixed
=emacs-ielm= entry points and helper functions rather than the raw map
variable.

** 3.7 =nelisp-emacs-special-buffers=

Residual rows are compatibility globals for scratch, messages, echo area,
and warnings buffers.  The stable prefixed API already covers the reusable
special-buffer behavior; the global names remain compatibility surface.

* 5. Extraction Implications

The current residual set does not block package extraction by itself.  It
does define follow-up constraints:

- Package archives should document prefixed stable APIs first and
  compatibility globals second.
- =emacs-*-builtins.el= files may remain in the package that owns their
  underlying behavior, but their helper names should not be advertised as
  consumer APIs.
- If a physical package split exposes =emacs-*-builtins.el= files, package
  README files must explain that they are compatibility layers.
- Future promotion work should not chase helper counts.  It should start
  from concrete consumer use cases and promote only the API shape that
  should be stable.

* 6. Gate Invariant

Libraryization work should keep this invariant:

#+begin_src text
promotion review queue: empty
stable API symbols: resolve after requiring nelisp-emacs
stable lazy API symbols: resolve after requiring their lazy feature
compat/helper/internal residuals: reviewed and documented
package verify failures: 0
library boundary report rows: 0
package dependency migration candidates: 0
#+end_src

If any =needs-*=, =*-review=, or =promote-ready= rows reappear, package
owners should handle those rows before claiming package extraction
readiness.
