* 0. Purpose

This document defines what an external project may rely on when it uses
=nelisp-emacs= as a reusable Elisp library.

The contract is intentionally smaller than the current repository.  The
goal is to let consumers reuse the pure-Elisp Emacs C-core replacement
surface without depending on =nemacs= application bootstrap, concrete
TUI/GUI frontends, or private helper names.

* 1. Contract version

The public facade exposes:

#+begin_src emacs-lisp
nelisp-emacs-library-contract-version
#+end_src

Current value: =1=.

Version 1 means:

- a consumer can add =src/= to =load-path= and =(require 'nelisp-emacs)=
  from =emacs -Q=;
- the facade provides the feature =nelisp-emacs=;
- reusable package groups can be discovered through the public query APIs;
- reusable package group loaders can be required directly from =emacs -Q
  -L src= without first loading the top-level facade;
- query results are copies, so consumers cannot mutate facade internals;
- requiring the facade does not load app/bootstrap, concrete TUI/GUI, or
  lazy editor files.

Increasing this version is required when the external consumer contract
changes incompatibly, for example when a package group is renamed, a query
API is removed, or the facade starts requiring a new bootstrap precondition.

* 2. Supported entry point

External consumers should load:

#+begin_src emacs-lisp
(add-to-list 'load-path "/path/to/nelisp-emacs/src")
(require 'nelisp-emacs)
#+end_src

Consumers should not require =emacs-init= unless they intentionally want
application bootstrap policy, vendor path setup, and app-facing lazy
loaders.

* 3. Public discovery APIs

After requiring =nelisp-emacs=, consumers may call:

| API | Contract |
|-----+----------|
| =nelisp-emacs-library-package-names= | return reusable package group names |
| =nelisp-emacs-library-package= | return a copied plist for one package group |
| =nelisp-emacs-library-package-features= | return copied member features for one package group |
| =nelisp-emacs-library-package-lazy-features= | return copied lazy companion features for one package group |
| =nelisp-emacs-library-package-manifest= | return a copied package manifest |
| =nelisp-emacs-library-stable-api-manifest= | return copied stable package API metadata |
| =nelisp-emacs-library-stable-api-symbols= | return stable package API symbols, optionally scoped to one package |
| =nelisp-emacs-library-stable-api-entry= | return stable package API metadata for one symbol |
| =nelisp-emacs-library-stable-lazy-api-manifest= | return copied stable lazy package API metadata |
| =nelisp-emacs-library-stable-lazy-api-symbols= | return stable lazy package API symbols, optionally scoped to one package |
| =nelisp-emacs-library-stable-lazy-api-entry= | return stable lazy package API metadata for one symbol |

The package plist contains:

| Key | Meaning |
|-----+---------|
| =:owner= | Doc 18 ownership group, such as =FND= or =BUF= |
| =:feature= | group loader feature required by the facade |
| =:features= | member features loaded by that group |
| =:lazy-features= | package-owned companion features kept out of facade eager loads |

Stable lazy API entries additionally contain =:feature=.  Consumers must
require that feature explicitly before relying on the listed symbols.
The stable lazy manifest may contain multiple entries for the same package
when a package owns more than one lazy companion feature; consumers should
use =nelisp-emacs-library-stable-lazy-api-symbols= and
=nelisp-emacs-library-stable-lazy-api-entry= instead of assuming package
keys are unique in the raw manifest.

The current public package names are =foundation=, =text-core=,
=buffer-core=, =editing=, =io=, =special-buffers=, =core=, and
=textmodes-stub=.  =text-core= exposes the reusable TXT regex substrate
and coding substrate through =emacs-text-core-features=.  =buffer-core=
intentionally names =nelisp-text-buffer=, =nelisp-emacs-compat=, and
=emacs-buffer= as explicit members so other packages depend on BUF
through the manifest instead of through hidden utility loads.  =editing=
owns the BUF editing commands and undo bridges separately from =core=,
which now depends on it as an explicit package dependency.
Mandatory package dependencies must appear either as same-package members
or explicit manifest package edges.  Lazy nested =require= forms should be
declared as =:lazy-features= when they are package-owned companions; these
appear as =lazy-manifest-package= in the generated dependency TSV.
Undeclared lazy reusable edges stay visible as
=lazy-unmanifested-reusable= for review.

The raw variable =nelisp-emacs-library-packages= exists for diagnostics,
but consumers should prefer the query APIs so the manifest shape can evolve
behind the facade.

interactive help history UI beyond the local back/forward stack,
mouse/widget/local help, xref source lookup, tutorial/distribution-file

=nelisp-ec-looking-at= use =nelisp-regex=.  Match data is a copied list of
1-based begin/end pairs; =nelisp-ec-match-beginning= and
=nelisp-ec-match-end= index that list by subexpression.

Buffer-local and default-value APIs are stable for NeLisp buffers through
=emacs-buffer-make-local-variable=,
=emacs-buffer-make-variable-buffer-local=,
=emacs-buffer-buffer-local-variables=, =emacs-buffer-buffer-local-value=,
=emacs-buffer-set-buffer-local-value=, =emacs-buffer-local-variable-p=,
=emacs-buffer-local-variable-if-set-p=, =emacs-buffer-default-value=,
=emacs-buffer-default-boundp=, =emacs-buffer-set-default=,
=emacs-buffer-setq-default=, =emacs-buffer-kill-local-variable=, and
=emacs-buffer-kill-all-local-variables=.  These functions use
=emacs-buffer='s side-table state and do not read or mutate host Emacs
symbol default bindings.  Callers that want NeLisp buffer-local writes
must use =emacs-buffer-set-buffer-local-value=; host =setq= interception is
outside contract version 1.  =emacs-buffer-make-variable-buffer-local=
records metadata used by =emacs-buffer-local-variable-if-set-p=, but it

Undo APIs are stable as an explicit record/apply layer through
=emacs-buffer-buffer-undo-list=, =emacs-buffer-buffer-disable-undo=,
=emacs-buffer-buffer-enable-undo=, =emacs-buffer-undo-boundary=,
=emacs-buffer-record-insertion=, =emacs-buffer-record-deletion=,
=emacs-buffer-undo=, and =emacs-buffer-modify-without-undo=.  Contract
version 1 does not auto-instrument =nelisp-ec-insert= or
=nelisp-ec-delete-region=; callers that want undo must record insertions
and deletions explicitly after mutating the buffer.  Undo-disabled buffers
store =t= as the undo-list sentinel and =emacs-buffer-undo= signals
=emacs-buffer-undo-disabled=.  Boundary markers are =nil= entries skipped
by =emacs-buffer-undo=.  Insertion records are =(BEG . END)= and undo by
deleting that range; deletion records are =(TEXT . POS)= and undo by
reinserting =TEXT= at =POS=.  =emacs-buffer-modify-without-undo= is a
dynamic helper that suppresses recording in its body and restores the prior
undo-list state on exit, including non-local exits.

The stable =editing= undo substrate exposes the same undo record model for

=emacs-undo-set-buffer-undo-list=, =emacs-undo-disabled-p=,
=emacs-undo-reset=, =emacs-undo-undo-boundary=,

=:after-success= hooks so GTK, TUI, and app commands can share the same
undo execution path.

Modification tracking APIs are stable through
=emacs-buffer-buffer-modified-p=, =emacs-buffer-set-buffer-modified-p=,
=emacs-buffer-restore-buffer-modified-p=,
=emacs-buffer-toggle-read-only-direct=,
=emacs-buffer-buffer-chars-modified-tick=,
=emacs-buffer-buffer-text-tick=, and
=emacs-buffer-bump-modified-tick=.  The modified flag is stored on the
NeLisp buffer record.  The chars-modified tick is a consumer-controlled
monotonic counter: callers bump it explicitly with
=emacs-buffer-bump-modified-tick= when they need cache invalidation for
non-text state.  The text tick is bumped automatically by the
=emacs-buffer= module after =nelisp-ec-insert=,
=nelisp-ec-delete-region=, and =nelisp-ec-erase-buffer=, and is not bumped
by text-property or overlay mutation.  =emacs-buffer-toggle-read-only-direct=
returns a plist containing =:read-only= and =:message= for frontend status
display.

Indirect buffer helpers are stable through
=emacs-buffer-clone-indirect-buffer= and
=emacs-buffer-buffer-base-buffer=.  An indirect clone shares the underlying
text storage with its base buffer, while point, narrowing, locals, undo
list, and extension state remain clone-local.

The low-level =text-buffer= substrate is stable through
=make-text-buffer=, =text-buffer-insert-char-code=,
=text-buffer-insert=, =text-buffer-delete=, =text-buffer-cursor=,
=text-buffer-set-cursor=, =text-buffer-substring=,
=text-buffer-search=, =text-buffer-length=,
=text-buffer-multibyte-p=, and =text-buffer-byte-length=.  This layer is a
display-free mutable text container with a cursor, byte/character length
queries, literal substring search, and insertion/deletion.  It deliberately
does not expose point, markers, overlays, text properties, undo, major
modes, or frontend state; consumers that need those semantics should use
the =nelisp-ec-*= and =emacs-buffer-*= stable APIs instead.

Redisplay-facing helpers remain outside the stable package API until their
consumer semantics are reviewed separately.  The stable narrowing contract
is limited to the NeLisp buffer substrate: =nelisp-ec-narrow-to-region=
sets the accessible half-open range =[START, END)=, clamps point into that
range, and leaves the underlying text storage intact; =nelisp-ec-widen=
clears that range.

* 4. Stability tiers

| Tier | Examples | Consumer rule |
|------+----------+---------------|
| Contract surface | =nelisp-emacs-library-contract-version=, facade query APIs, stable package API query APIs, stable lazy package API query APIs | external consumers may rely on this when the library gate is green |
| Stable package API | symbols returned by =nelisp-emacs-library-stable-api-symbols= | reusable package API accepted into the facade contract and available after requiring =nelisp-emacs= |
| Stable lazy package API | symbols returned by =nelisp-emacs-library-stable-lazy-api-symbols= | reusable package API accepted into the facade contract but available only after requiring the entry's lazy feature |
| Reusable public API | prefixed rows in =make nemacs-public-api-inventory=, such as =emacs-buffer-*= | candidate public surface; stable only when the owning package documents, tests, and adds it to the stable API manifest |
| Compatibility global | =current-buffer=, =find-file=, =dired= style shim names | use when Emacs-compatible behavior is the goal; prefer prefixed APIs for new integrations |
| Adapter/app API | GUI/TUI/backend/app-owned symbols | not part of the reusable package contract |
| Private helper | names containing =--= | never use from external consumers |

Doc 28 records the stable API compatibility policy.  Current generated
coverage is 948 stable API rows: 935 =public-prefixed= symbols and
13 =compat-global= symbols, with 0 policy failures.  A =compat-global=
symbol is stable only when it appears in the facade stable eager or stable
lazy API query results.  Other Emacs-compatible global shim names remain
advisory until promoted into the stable manifest.

* 5. Non-goals

Version 1 does not promise:

- package installation metadata;
- a final package directory layout;
- semantic versioning for every prefixed symbol;
- compatibility with requiring files outside =src/=;
- availability of =nemacs-main=, =emacs-init=, =gui/=, or runtime image
  bootstrap;
- that every Emacs-compatible global shim is safe to install into an
  arbitrary host Emacs session.

* 6. Verification

The focused consumer gate is:

#+begin_src sh
make nemacs-library-consumer-smoke
#+end_src

It runs a fresh =emacs -Q -L src= process, requires =nelisp-emacs=, checks
the contract version and query APIs, and verifies that app/bootstrap,
concrete TUI/GUI, and lazy editor files were not loaded.

The contract symbol gate is:

#+begin_src sh
make nemacs-library-contract
#+end_src

It verifies that the stable consumer contract symbols exist after requiring
=nelisp-emacs= and writes the narrow contract TSV/Org artifacts.

The stable API compatibility policy gate is:

#+begin_src sh
make nemacs-library-compat-api-policy
#+end_src

It verifies that stable eager and stable lazy API symbols are either
=public-prefixed= preferred library API or intentional =compat-global=
Emacs-compatible shim API.  It fails if a stable manifest promotes a private
helper, an app/frontend adapter, or a symbol missing from the generated public
API inventory.

The package group gate is:

#+begin_src sh
make nemacs-library-package-smoke
#+end_src

It runs each reusable package group loader in its own =emacs -Q -L src=

* 7. Change rules

- Add new reusable files to Doc 18 ownership before exposing them through
  the facade.
- Add or update group loader =*-features= manifests when package
  membership changes.
- Keep package group loaders directly requireable from =emacs -Q -L src=.
- Keep facade package dependency =app-or-frontend= rows at zero.
- Keep the consumer smoke green for facade membership changes.
- Update Doc 24 public API baseline when public symbols are added or
  removed intentionally.
- Bump =nelisp-emacs-library-contract-version= for incompatible external
  consumer contract changes.
- Do not promote GUI/TUI/app convenience behavior into the contract unless
  it is first owned by a reusable package group.
