* §2 タスク本体: char-table substrate 実装

** 2.1 必要 op（isearch-mode-map 実走で確定した最小集合）
1. =(char-table-p OBJ)= → char-table なら t。
2. =(make-char-table SUBTYPE &optional INIT)= → char-table を生成。
3. =(set-char-table-range CT RANGE VAL)= / =(char-table-range CT RANGE)=
   - RANGE = nil(default) | t(全体) | =(FROM . TO)=(範囲) | CHAR(単一)。
4. =(aref CT CHAR)= / =(aset CT CHAR VAL)=（char 添字アクセス）。
5. =(map-char-table FUNCTION CT)=（非 default slot を (KEY . VAL) で funcall）。
6. =(char-table-parent CT)= / =(set-char-table-parent CT NEW)= / =(char-table-subtype CT)=。
7. =(max-char)= → =#x3FFFFF=（Doc 05 = UTF-8 / Unicode max）。

** 2.2 char-table の表現（推奨）
records が標準で使えるか不明なら *tagged vector* を使う（vector op は確実に存在）:
#+begin_example
  [--nemacs-char-table SUBTYPE DEFAULT PARENT ASCII-VEC RANGES]
    slot0 = tag symbol '--nemacs-char-table（char-table-p の判定キー）
    SUBTYPE  = make-char-table の引数
    DEFAULT  = 範囲外/未設定 char の既定値
    PARENT   = 親 char-table または nil
    ASCII-VEC = (make-vector 256 nil)  ; char<256 の高速 slot
    RANGES   = alist: ((FROM . TO) . VAL) ...  ; char>=256 の疎な範囲（load 用は格納のみで十分）
#+end_example
- =char-table-p= = =(and (vectorp x) (> (length x) 6) (eq (aref x 0) '--nemacs-char-table))=。
  （注: 真 Emacs では char-table は vectorp で nil だが、load を通すには tagged vector で実用上十分。
   将来 reader primitive 化する際に独立型へ。）
- =aref/aset= は CHAR<256 を ASCII-VEC、>=256 を RANGES 走査（無ければ DEFAULT/PARENT）。
- lookup の厳密性は *runtime の課題*。load を通すには「格納が落ちない」ことが最優先。

** 8.6 org-cycle.el hard-stop の正体 = 累積メモリ枯渇（2026-06-05 特定済）
§4 のエラー tag dump は *捕捉される Lisp error* 用で、この hard-stop（reader が exit 0 で終了）には不適。
*直接 reader 実行 + truncation 二分 + /proc VmHWM 測定*で特定:
- *不足 primitive でも私の char-table/keymap 変更でもない*:
  - 停止フォーム =org-cycle-show-empty-lines= は header+bootstrap のみの隔離で *正常に定義*（exit 55 生存）。
  - メモリ増加は *フォーム数に一様比例*（isearch=keymap-heavy も simple.el も charscript も同程度/行）→ keymap 固有でない。
- *正体 = 累積メモリ枯渇*:
  - bootstrap のみで *1.3GB*、vendor 1-42 + org-cycle 大半で *13.9GB*。
  - reader GC: 新 threshold = =max(3×live, bump+512MB)= → *RSS ≒ live set の 3 倍*。
    reader は *"no interning"*（=bf_eq2= コメント: symbol は name 比較、intern 無し）= 同名 symbol の
    出現ごとに新規 symbol+name を割当 → 319 file の定義群で live set が肥大。
  - *物理 RAM(64GB)ではなく reader の arena/allocator 上限 ~14GB* に達し alloc 失敗 → exit 0。
    org-cycle.el (#43) は累積使用量が上限を超える地点にすぎない（特定フォーム依存でない）。
- *結論*: これは *reader(dev/nelisp)のメモリ効率問題*。nemacs substrate primitive では解けない。
  真の対策は reader 側の *symbol interning / s-expr 共有*（live set 縮小）。GC headroom 3× や
  threshold 512MB を下げるのは doc §6 通り thrash するので非推奨。arena 上限引き上げは延命のみ。
- *診断手順の証跡*: =/tmp/full-prog.el= を保存し =build/nelisp-experiment --load= を直接実行、
  =head -n N= で truncation 二分、=/proc/PID/status VmHWM= で peak RSS 測定（replay の出力捕捉編集は revert 済）。

** 8.7 メモリ blowup の真因と修正設計（2026-06-06 深掘り）
§8.6 の「累積メモリ枯渇」を reader 内部まで追い、真因と修正方向を確定:

*** 真因1: GC トリガが総使用量を見ていない（FIXED, sound）
- GC トリガ（=bf_eval_source_string_loop= 等 3 箇所）が =268435456=（*チャンク0 bump、256MiB cap*）を
  threshold(512MiB) と比較 → Doc 140 のチャンク成長化後、あふれ分は新チャンクへ行き bump は 512MiB に
  *永久に届かず → tracing GC が一度も発火しない*（=live-after-gc=0= を 12.7GB まで確認）。
- *修正*: トリガ/threshold-update を =268436184=（*総 chunk-bytes-reserved*）へ。3 箇所。
  これは sound（=(+ 40 2)= 等 + load-smoke + heavy load で segfault なし）。メモリ 13.9GB→10GB。

*** 真因2: free-list reuse が exact-fit 単一リスト線形スキャン（thrash + reuse 非効率）
- =nl_freelist_take= は単一リスト(=268435552=)を *exact BLOCK_TOTAL 一致* で線形スキャン。
  GC sweep が数千の混在サイズブロックを free すると、*可変長 alloc（symbol-name/string、各長さ違い）が
  固定長 cons/Sexp ブロックを毎回全走査* → O(alloc数 × free-list長) 二次関数。
  GC を有効化すると load が *~20-30× 遅く*なる（bootstrap だけで >200s）。
- *さらに reuse 非効率がメモリにも効く疑い*: exact-fit 不一致で可変長 alloc が reuse できず、
  freed ブロックが free-list に滞留＋*OS 未返却*（チャンクは munmap されない）→ =used=（bump 累積）が成長、
  RSS=used。つまり 10GB の多くは *「freed-but-not-reused-nor-returned」* で、reachable retention とは限らない。
  → *正しい segregated free-list で速度とメモリの両方が改善する可能性*（eval-escape より単純）。

*** 修正設計: size-segregated free-list（O(1) exact-fit）
- BLOCK_TOTAL 別にバケット化。バケット配列は reserved 領域の *空きギャップ base+240..+696*
  （=268435696..268436152=、grep で未使用確認、mmap-zero なので init 不要）。
  bucket head = =268435696 + (BT-24)= for BT in [24,480]、BT>480 は fallback 単一リスト(=268435552=)。
- *重要な落とし穴（私の試作が segfault した原因）*: =nl_gc_bt_ok= は BLOCK_TOTAL>=16 を通すが、
  bt=16（walker desync 等）で =(+ 268435696 (- 16 24))= = =268435688= = *base+232（使用中スロット）= wild write* → 破損。
  → *bt<24 も fallback へ落とすガード必須*: =(if (or (< bt 24) (> bt 480)) 268435552 (+ 268435696 (- bt 24)))=。
  最初の試作はこのガードが無く非決定的 segfault（heavy load 時）。revert 済み。
- バケット化が正しく入れば load は *fast* になり、その状態で =268435640=（survive）/=268435632=（freed）
  カウンタや live-after-gc を読めば、残りメモリが reachable retention（eval-escape）か reuse 非効率かを確定できる。

*** 残るかもしれない eval-escape（真因3 候補、未確定）

  ただし現行 =nl_cons_macro_apply_eval= は *in-place cache せず毎回再展開*（exp_slot は per-call scratch）で、
  in-place macro caching は現設計に見当たらない。eval-escape が残差として効くかは、
  segregated free-list で fast 化してから survive/freed 比で測るのが正道。

** 8.8 最終診断: residue と segfault は同一根本原因 = GC mark の不完全さ（2026-06-06）
§8.7 で「segregated free-list で reuse を効かせれば memory も下がる」と見積もったが、*実装すると segfault*。
原因はバケットコードではなく、より深い所にあった:

- *単一リスト exact-fit reuse は実質ほとんど reuse が起きない*（可変長 alloc がサイズ不一致で毎回 bump）。
  → GC が free した block の多くは *再利用されず放置* = メモリに滞留（これが §8.6/8.7 の「residue ~74%」の正体）。
- *segregated free-list は reuse を効かせる* → free 済み block を積極的に再配布。
  ところが *tracing GC の mark が不完全*（eval-escape: aliasing clone / mirror-closure sharing で
  reachable になった box を mark しきれない）ため、*まだ生きている block を sweep が free している*。
  単一リストでは「free された生 block」が再利用されず放置されるだけ（UAF にならない、メモリが膨らむだけ）。
  segregated で再利用すると *use-after-free → segfault*。
- *結論*: residue（メモリ滞留）と segfault は *同一根本原因の二症状* = GC mark の不完全さ。
  - GC を走らせない（元の状態）= sound だが 14GB blowup。
  - GC を走らせる（GC-trigger fix）= truly-unreachable を回収して 14GB→10GB、reuse がほぼ無いので UAF は顕在化せず *実用上 sound*。だが残り 10GB は「mark 漏れで free されたが再利用されない生 block + 真の live」。
  - GC を走らせ *かつ reuse を効かせる*（segregated）= mark 漏れが UAF として露呈 → 使えない。
- *安全に memory を下げる前提条件*: *GC mark を sound にする*（escape した box を全て mark = roots と
  traversal の完備）。これは NOTE(2026-06-01) の「correct fix is tracing mark-sweep GC at the form boundary」
  の *mark 側の完備* に相当し、eval-escape の本丸。深い eval/GC correctness 課題。

*** 採用した最終状態（sound 優先）
- *GC-trigger fix のみ採用*（3 箇所、=268436184= 総 reserved でトリガ）。GC 稼働・14GB→10GB・UAF 未顕在。
  ただし free-list thrash で *遅い*（reuse がほぼ無いので exact-fit scan は短く済むはずだが、
  GC mark/sweep が毎回全 block を walk するコスト + GC 頻度で遅延）。
- segregated free-list は *revert*（unsound GC を露呈するため、mark 健全化まで封印）。
- 次にやるべきは *GC mark の完備*（eval-escape box を漏れなく mark）。それが入って初めて
  reuse 効率化（segregated free-list）と低メモリが *両立* する。

** 8.9 GC mark 不健全の設計上の根本原因 = refcount × tracing-GC の不整合（2026-06-06 設計調査）
§8.8 の「GC mark が生 box を free → UAF」を reader 内部まで追い、設計レベルの真因を確定:

*** reader は hybrid: refcount + tracing-GC
- *box は refcount を持つ*: =nelisp_nlconsbox_clone= = =(atomic-fetch-add (+ box-ptr 64) 1)= →
  *rc は box+64*。clone は rc を bump、=nl_sexp_clone_into= は "refcount-aware copy"、
  =nl_vector_set_slot= は "refcount-aware slot write"、NlBox Drop kernel は "if-zero-refcount free"。
  = eval 機構は box の寿命を *refcount で管理* している。
- *一方 tracing GC（=nl_gc_sweep_one=）は mark==0 を free / mark==1 を survive — refcount を一切見ない*
  （=nl_gc_mark_slot= は Cons7/Vec8/Rec12/Cell11/MutStr6/Str5/Sym4 を reachability で mark）。

*** 不整合 = UAF と residue の両方の根本原因
- clone が作る *refcount-alias*（rc>0、生きている）が GC root から到達不能だと、
  tracing sweep は mark==0 として *free* → alias-holder が dangling → *use-after-free*。
  NOTE(2026-06-01) の「refcount-aliasing clones escape the mark」はこれ。
- 単一リスト exact-fit は reuse がほぼ無く、free された生 box が再利用されず放置 → residue（メモリ滞留）。
  segregated で reuse を効かせると同 box が再配布され UAF が顕在化 → segfault。
- = *両症状は refcount×tracing の二重管理不整合* に帰着。S0 "sound GC" は *一度も発火していなかった* ため
  この不整合が露呈しなかっただけ（§8.7）。

*** 修正設計の選択肢
1. *refcount-aware sweep*（最小・sound）: =nl_gc_sweep_one= を *mark==0 かつ rc==0 の時だけ free* に。
   生 box（rc>0）を絶対 free しない → UAF 消失。reuse（segregated）も安全に効く → 低メモリ両立。
   - 課題: rc offset が box 型ごとに異なる（cons=box+64、Sym/Str は別位置）。sweep は header walk で
     型を知らないので、*型別 rc-offset の特定* or *全 box 型で rc を統一 offset 化* が必要。
     rc 漏れ cycle（rc>0 だが到達不能な循環）は free されず leak だが、eval garbage では稀。
2. *pure tracing GC（refcount 撤廃）*: clone/slot-set/drop を refcount-free な pointer copy に統一し、
   寿命を tracing GC 単独に。不整合は消えるが、tracing mark の *roots/traversal を完備* する必要があり、
   refcount ロジック除去は広範。クリーンだが大規模。
3. *pure refcounting（tracing 撤廃）* + cycle 対策: 元の per-eval reset は escape で unsound だったので非推奨。

*** 推奨と次の一手
- *推奨 = 選択肢1（refcount-aware sweep）*。sound・小変更・低メモリ/高速/健全の三立に最短。
  前提作業 = *全 box 型の rc-offset を確定*（cons=+64 は判明。Symbol/Str/Vector/Record/Cell/MutStr の
  rc 位置を NlBox Drop kernel =lisp/nelisp-cc-alloc-dealloc.el= 等から洗い出す）。統一できれば sweep に
  =(if (= (ptr-read-u64 (+ obj RC_OFF) 0) 0) free keep)= を 1 行足すだけ。
- これが入れば segregated free-list（§8.7、bt<24 ガード付き）を安全に再投入でき、メモリ大幅減が見込める。

** 8.10 refcount-aware sweep を実装・検証 → 効かず。本丸は mark traversal の不完全さ（2026-06-06）
§8.9 の設計（refcount-aware sweep）を実装して検証した結果と、修正された理解:

*** 実装した内容（検証後 revert 済）
- *rc-offset survey*: clone kernel から各 box 型の rc bump offset を確定 ―
  cons=+64 / record=+56 / cell=+32 / vector=+24 / boolvector=+24 / str=+24 / chartable=+120。
  *いずれも =BLOCK_TOTAL-24=（= block の最後 8 バイト = =hdr+BT-8=）に一致*（cons BT=88→64、
  str BT=48→24、record BT=80→56…）。→ *sweep は型を知らずに rc を一意に読める*。
- =nl_gc_sweep_one= を *mark==0 かつ rc(=ptr-read hdr+BT-8)==0 の時だけ free* に変更（refcount-aware）。
  segregated free-list（§8.7、bt<24 ガード）を rc-aware sweep の上に再投入して r-prog を実行。

*** 結果: *まだ segfault*（rc-aware sweep は UAF を解消しない）
- = 誤って free される生 box は *rc==0*。refcount-alias ではない。
  → 本丸は *mark の TRAVERSAL が reachable な edge を辿りきれていない*（roots 不足ではなく辿り漏れ）。
  refcount × tracing の不整合（§8.9）は二次的で、主因ではなかった。
- correctness（=(+ 40 2)= 等 + load-smoke）は rc-aware/segregated 単体では通るが、
  heavy load + 効果的 reuse で mark 漏れ box が再配布され UAF。

*** mark traversal の漏れ候補（次の調査対象）
- =nl_gc_mark_slot= は Cons7/Vec8(slots 0..len)/Record12(slots 0..len)/Cell11/MutStr6/Str5/Sym4 を walk。
- 漏れ得る edge:
  - *mirror の bucket chain*: globals_record のバケットが entry を辿る "next" を、mark が walk しない slot
    （Record の len を超える領域 等）に持つと、bucket 先頭以外の entry とその子が mark 漏れ。
  - *Record/Vector の len を超えるスロット*（rc やメタを末尾に持つ型で、子 Sexp が len 外にある場合）。
  - *error/throw stash*（268435480/268435512）は root ではない。
- = *次の一手 = mark の完備性監査*: 各 box 型の「実際の子 Sexp 位置」と mark_slot の walk 範囲を突合し、
  辿り漏れている edge を特定して mark_slot/roots に追加する。これが入れば segregated free-list を
  安全に再投入でき、低メモリ・高速・健全が三立する。

*** 最終状態（このセッション）
- *採用 = GC-trigger fix のみ*（sound・14GB→10GB・thrash で遅い）+ user の ptr-call（未コミット同居）。
- segregated free-list / refcount-aware sweep は *検証の結果 revert*（mark 未完備のため封印）。
- =build/nelisp-experiment= = sound な GC-trigger-fix のみ。

** 8.11 mark 完備性監査の結果（2026-06-06, read-only）
§8.10 の「mark traversal が辿り漏れ」を監査し、主犯候補を除外:

*** 監査済み = 完全（主犯ではない）
- *mirror（globals_record）の traversal は完全*。構造（=mirror-bucket-prepend.el= より）:
  =mirror Record → slot0=HT Record → slot1=buckets Vector → 各 bucket = Cons chain
  (box.car=(KEY-Str . ENTRY-Record), box.cdr=next cell)=。
  =nl_gc_mark_cons= は *cdr-spine をループ*で辿る（car を再帰）ので *bucket chain 全体*を mark し、
  entry Record の value/function/plist/constant を Record walk（slots 0..len）で mark。→ 漏れ無し。
- *標準 box 型*（Cons7/Vec8/Record12/Cell11/MutStr6/Str5/Sym4）は =nl_gc_mark_slot= が
  各々の子 Sexp を walk。Record は type_tag(box+0) + slots(data_ptr[0..len]) を mark。→ 漏れ無し。
- *バケット領域 base+240..+696 は mmap MAP_ANONYMOUS(flags=34) = カーネル zero-fill 保証* →
  未初期化 garbage ではない（segregated segfault の別仮説も棄却）。

*** 残る容疑（read-only で確定不可、runtime debugging が必要）
1. *frames(ctx+32) / closure の env キャプチャ*: closure が捕捉した lexical env を、mark が辿らない
   位置に持つ場合。frame stack の entry レイアウトと mark 範囲の突合が未了。
2. *error/throw stash(268435480 tag / 268435512 val) が GC root でない*: form の eval が stash を
   残すと、その box が mark 漏れ。
3. *segregated free-list 自体の構造バグ*: read-only では nl_freelist_take/free_block の論理は
   single-list と等価に見えるが、heavy load 時のみ segfault。

*** 次の決定的手段 = runtime poison
- mark/sweep を一時計装: =nl_gc_free_block= で payload を毒値(例 0xdeadbeef)で埋め、
  =nl_gc_mark_slot= / box-access で毒値検出 → *実際に UAF する box の型と出所を捕捉*。
  または GC を mark-only(=268435584=1=)にして free を止め、segregated でも segfault するか切り分け
  （しなければ free/reuse 起因=UAF 確定、するなら segregated 構造バグ）。
- これは reader instrumentation（read-only 範囲外）。次セッションで実施し、漏れ box を mark_slot/roots に追加 → 
  segregated free-list 安全再投入 → 低メモリ達成。

** 8.12 runtime 切り分けの結論: GC mark は健全。segfault は segregated reuse の subtle bug（2026-06-06）

*** 切り分け手順と結果

- = 低メモリ化の前提は「GC mark の健全化」ではなく *「segregated free-list の reuse bug を gdb で特定」*。
  mark は既に健全なので、その 1 点さえ直せば segregated（O(1) reuse）で *低メモリ・高速・健全が三立*する。

** 8.13 訂正: segregated free-list は健全だった。segfault は診断プローブの artifact（2026-06-06）

  reuse で回収不能（live だから）。低メモリ化は *eval-escape / over-retention（reachable な多重 clone /
  macro 展開保持）* の調査が必要 = §8.6/8.9 の residue 課題（深い、別途）。

*** 採用した最終状態
- *GC-trigger fix + size-segregated free-list を採用*（両方 sound、real load exit 0 @35s、14GB blowup 解消）。
  + user の ptr-call（未コミット同居）。=build/nelisp-experiment= = この reader。
- *CAVEAT*: segregated は latent な GC root-coverage gap に「less forgiving」。診断系コード（arena-stats を
  list/repr する等）は GC root に乗らない中間値で corruption しうる。real load は無問題だが、留意。
- 低メモリ（10GB→）の本丸 = over-retention（residue ~67x）。gdb 等が使える環境で eval の reachable
  多重保持を追うのが次。

- = 残りは *over-retention でなく genuine な loaded 構造*。だが NeLisp の *1 node = 48B*（16B header + 32B Sexp）、
  *cons = 88B* で、実 Emacs（tagged value 8B / cons 16B）の *約 3-5 倍重い表現*。§8.13 の「genuine ~60MB」は誤り。

*** 数百MB級への道（深い core 変更、いずれも多段・要慎重）
1. *GC header 削減*（最も isolated、~480MB 見込み）: 各 box の 16B header(BLOCK_TOTAL u64 + mark u64) を 8B へ
   （mark を BLOCK_TOTAL の低ビットへ pack、or 別 bitmap）。28M box × 8B = ~230MB 減。allocator+GC の全 box
   アクセスを触るが論理は局所。*第一歩として最有力*（validate しやすい）。
2. *immediate の inline tagged 化*: int/nil/t を 32B Sexp box でなく tagged value(8B)に。Sexp 表現の再設計 = 広範。
3. *Sexp packing / 16B cons*: cons を car/cdr の 32B inline Sexp(72B) から tagged-pointer pair(16B)へ。最大効果だが
   表現全体の作り直し（reader/eval/GC/clone 全経路）。
4. *byte-compile*: source AST を保持せず compact bytecode へ（実 Emacs 方式）。最も Emacs 的だが別サブシステム。
- 推奨順: まず (1) GC header 削減で ~230MB を validate（リスク中、4438 で回帰確認）→ 効果と手応えを見て (2)/(3)/(4)
  の本格再設計へ。pool(over-retention) と interning(重複) は完了済、残りは「genuine だが重い表現」の本質課題。

  census exploration として実施・revert 済（byte 累積に bug 出たが COUNT は信頼可、28M BT48 を確定）。

*** 8.18.1 GC header 16B→8B = SHIPPED（commit 7134b383、2.86GB→2.60GB、4438 PASS）
(1) GC header 削減を実装・着地。バグ 2 つを潰して完成:
- *設計*: box header を BLOCK_TOTAL(8-aligned 高ビット) + mark(低 3 ビット) の単一 u64 に。obj = hdr+8（was +16）。
  ヘルパ =nl_hdr_bt/nl_hdr_mark/nl_hdr_set_mark=（低 3 ビット mask、負数 logand 回避で減算抽出）。
- *変更サイト（全て standalone-build.el の allocator/GC に局所、外部ユニットは obj 相対で不変）*:
  nl_block_total(8), nl_chunk_try_alloc(obj=old+8 / BT@obj-8 / mark 別書込削除), nl_freelist_take/scan(sentinel
  クリア + BT@cur-8 masked), nl_gc_mark_block(mark 低ビット), nl_gc_free_block(_link)(mark=2 / next@hdr+8),
  nl_gc_sweep_one/step(mark/BT masked), nl_alloc_zero_fill 呼出 (- want 16)→(- want 8)。
- *バグ1*: =nl_alloc_zero_fill= の reuse ゼロ埋めが旧 payload=want-16 前提 → 末尾 8 バイト stale。→ want-8 へ。
- *バグ2（hang の主因）*: 8B 化で全 BT が −8 シフト（最小 BT 24→16）したのに、segregated free-list の bucket が
  依然 =BT-24= / 範囲 [24,480] のまま → 小ブロック(BT16-23)が *線形 fallback list に落ち O(n²) thrash* し、
  dired.el(#, line 31159) の GC で stuck（hang に見えた）。→ bucket を *[16,472] / index BT-16* に再校正
  （nl_freelist_take + nl_gc_free_block）。これで小ブロックが bucket に戻り解消。
  *特定法*: gdb 不要だった = 「BT シフトで bucket 校正がズレた」と reasoning で当て、再校正して完走で確認。
  （gdb は安全網としてインストール済 = 万一 reasoning が外れたら attach→backtrace の予定だった。）
- *結果*: full vendor-load *2.86GB→2.60GB(-260MB)*、~11s、hang 無し、org-cycle 停止（天井は不変）。
  =make test-parallel= *4438/0 unexpected*（nelisp-process sentinel は並列負荷の env flake、単独 47/47）。eq 正常。
- *次の表現変更（数百MB へ続く）*: §8.18 の (2)immediate inline tagged / (3)16B cons / (4)byte-compile。

  （immediate を tagged 8B 値へ）が次の大物。

*** 8.18.2 gdb 調査メモ（バイナリは not stripped・887 sym、backtrace 動作可）

  - vector(BT40 box): =nl_logic_build_scratch=（=nl_alloc_vector 11=、let束縛ごとの 11-slot frame entry）←

  - bare Sexp(alloc-bytes 32): =nl_apply_sym_eq_bytes=（builtin 名比較）← apply_builtin ← eval_inner_cons。eval の
    一時 scratch。
- *教訓*: 確保レートサンプリング（gdb bt）は *頻出 transient scratch* を拾い、*retained 28M* は拾えない
  （alloc-rate ≠ retention）。retained 28M bare Sexp（census: obj+8 小値）の正体特定には *live-heap 解析*が要る:
  - 案: nelisp--size-census を拡張し BT40 boxes の obj+0 tag / obj+8(ptr-like vs 小) を *正しく* 集計（前回 byte

*** 8.18.3 gdb live-heap walk で 28M 箱の正体確定 = lexframe 束縛エントリ（over-retention 疑い）
gdb python で深い GC 地点（reserved=1.07GB）の arena を walk（=/tmp/heapwalk.gdb=、nl_gc_collect で break →
nl_arena_base(0x54b000) から chunk/block walk、BT40 live box を 2.5M block 分 sample）:
- *BT40-live*: bare Sexp(obj+8 小値) *1,503,447* / vector(obj+8 ptr) 157,049 → *bare Sexp が 90%*。
- *tag 分布*: Nil(0) 713K + Symbol(4) 747K + Cons-ptr(7) 57K + T(1) 40K + 他。
- *決定的サンプル*: =(tag4, cap12, ptr0x531599, len12)= が *同一 name ptr で 40 中 ~15 回反復*（~37%）=
  同じ 12 文字シンボルを *別々の Sexp box で大量保持*。これは =nl_logic_write_symentry=
  （=lisp/nelisp-cc-evalport-env-leaves-logic.el=）が各 let 束縛で書く *symentry シンボル*。=(Nil,0,0,0)= の
  standalone Nil box も多数。
- = *28M 箱の正体 = lexframe の束縛エントリ構造*: =nl_logic_bind_local= → =nl_logic_build_scratch= が束縛ごとに
  11-slot vector + symentry シンボル box(sym_slot) + value/unbound clone を作る。symentry box が *3M+* 個 =
  グローバル ~50K を遥かに超える → *局所(let\*)束縛のエントリが解放されず累積する over-retention の疑いが濃厚*
  （genuine AST ではない）。pool 肥大と同型の「transient のはずが保持」パターン。
- *次の一手（pool 修正級にトラクタブルな可能性）*: =nl_logic_bind_local= / frame push-pop / =nelisp_frame_pop= が
  束縛エントリ(11-slot vector + sym_slot box)を *scope 退出時に解放しているか* を確認。していない（or GC root から

*** 8.18.4 *真の発見*: 「over-retention」ではなく GC 間のゴミ蓄積。live set は極小（commit fa65d14e）
=/tmp/heapwalk2.gdb=（=nl_gc_sweep= entry = mark 後・sweep 前で break、BT40 を mark 別に分類）で確定:
- *2.5M sample 中 BT40: LIVE(mark1)=109 のみ / GARBAGE(mark0)=1,660,387*。= 先に「保持」と見えた symentry/Nil/
  Symbol 箱は *全て回収可能なゴミ*（この GC で sweep される）。*live set は極小*（数十MB 程度）。
- = 2.6GB の peak は *over-retention でなく per-eval ゴミの GC 間蓄積*。トリガが reserved+512MB 再arm ゆえ、
  各 GC 前に ~0.5GB のゴミが溜まる。lexframe 束縛エントリ（11-slot vector + symentry box + value clone）が
  per-binding に生成され、scope 退出後ゴミ化するが、次の tracing GC まで回収されない（per-eval LIFO reclaimer は
  §8.x で unsound として無効）。
- *修正 = GC トリガ 512MB → 64MB*（init + re-arm 3 = 4 site、=536870912→67108864=）。mark は live 極小で安価ゆえ
  8x 頻発でも +3s（11→14s）。*full load peak 2.60GB → 1.80GB（決定的・4 run 安定）*。64MB は boot(~60MB)直上で
  GC-during-boot 回避（16MB は boot 中 GC で segfault、32MB は boot 天井未満ゆえ不可）。4438/0、eq 正常。
  *注*: =536870912= は trigger 専用で衝突無し。=67108864=(64MB) は chunk-size/intern-region でも既出だが別用途。
  =16777216=(intern table size) / =33554432= 等を sed 一括置換すると intern region を壊す（segfault）— 個別に。

*** 8.18.5 数百MB級への残りの道（live 極小ゆえ射程内）
peak 1.80GB の内訳: *live 極小 + GC 間ゴミ(~64MB window) + 解放済だが munmap されない reserved chunk の累積*。
reserved(mmap chunk) は *縮まない*（munmap しない）ので peak = 到達した最大 reserved。さらに下げる手:
1. *空チャンクの munmap*（最有力・live 極小ゆえ効く）: sweep で全 free になった chunk を OS へ返す。per-chunk
   free-count 追跡 + free-list から該当 block を外す + chunk list から unlink + munmap。中規模 GC 変更。live が
   数十MB に集中するなら peak が *数百MB* まで落ちる見込み。
2. *boundary-reclaim 健全化*: per-eval LIFO reclaimer を sound に再実装し、束縛エントリ等を form 境界で即回収
   （ゴミが arena に積まれる前に）。eval-escape correctness 課題（§8.9）と絡む、深い。
3. トリガ更に下げる: root-coverage gap（§8.13）露出で 16MB segfault。gap 健全化が前提。
- *推奨 = (1) 空チャンク munmap*。live 極小という今回の発見が効く。gdb で sweep 後の chunk 占有率を見て設計。

*** 8.18.6 空チャンク munmap を試行 → fragmentation で不発（revert、compaction が必要）
(1) 空チャンク munmap を実装・試行（実装後 revert 済、fa65d14e が HEAD）:
- *実装*: nl_gc_sweep を「free-list clear → 各 chunk で livecount → 空(live=0)かつ非persistent かつ非current
  なら munmap(syscall 11)+unlink+reserved/chunk-count 減、それ以外は dead を free-list 再構築」へ。clear-first で
  munmap chunk の block は再追加しない（dangling 無し）。build/sanity/full-load(rc=0) OK。
- *結果*: *peak RSS 不変(1.76GB)・遅い(14→20s)* = 効果無し。原因 = *fragmentation*。
  live set は極小(~109 BT40)だが *全 growth chunk に散在*（各 chunk に数個の live ブロック）→ *完全に空の chunk が
  ほぼ生じない* → munmap がほとんど発火しない。加えて VmHWM(peak) は historical max ゆえ、仮に current RSS を
  下げても peak metric は不変。
- *壁 = 非moving GC の限界*: 散在 live が全 chunk を pin。chunk-0(256MB persistent) + pin された growth chunk で
  peak ~1.76GB。
- *数百MB への真の道（いずれも大規模）*:
  1. *compacting(moving) GC*: live を少数 chunk へ relocate → 残り chunk を空にして munmap。全ポインタ(cons car/cdr,
     vector slot, mirror, frame)の更新が必要 = 大手術。live 極小ゆえ relocate コストは小さいはずだが実装は重い。
  2. *generational / live-vs-transient 分離 alloc*: retained と per-eval transient を別 region へ。transient region は
     form 境界で一括 reset(boundary-reclaim 健全版)。fragmentation 回避。
  3. *sound boundary-reclaim*(§8.9 eval-escape): per-eval garbage を form 境界で即回収 → arena を汚さない。深い。
- = munmap 単体は不可。*compaction か transient 分離*が前提。1.80GB(64MB trigger) が non-moving GC の実用的下限。
