* 0. Purpose

This inventories the 223 =app-gui-semantic-definition= rows reported by
=make nemacs-library-boundary-report= after private cross-owner edges were
reduced to zero.

These rows are advisory.  They do not all mean the implementation is in
the wrong place.  The report intentionally over-matches names that look
like reusable editor semantics inside APP/GUI-owned files.  This document
separates:

1. reusable runtime adapters that are currently classified as GUI,
2. app/TUI command semantics that should move into shared packages,
3. concrete GTK/transport behavior that should remain frontend-owned, and
4. report noise that needs better ownership metadata before enforcement.

* 2. Inventory classes

** 2.1 Shared GUI command runtime adapters

Files:

- =src/emacs-fileio-gui.el= (54)
- =src/emacs-dired-min-gui.el= (45)
- =src/emacs-help-gui.el= (23)

Total: 122 rows.

These files are not ordinary frontend code.  They define bridge-friendly
command runtimes that can be consumed by GTK, TUI, and baked standalone
images through registered backend callbacks.

Examples:

- =emacs-fileio-gui-command-spec=
- =emacs-fileio-gui-writeback-spec=
- =emacs-dired-min-gui-writeback-spec=
- =emacs-help-gui-describe-key-core=

Decision:

- Keep these as reusable runtime adapters, but do not treat them as
  concrete GUI owners.
- In Doc 18, their primary owner is their semantic package first, with
  GUI as a secondary adapter facet:
  - =emacs-fileio-gui.el=: =IO/GUI=
  - =emacs-dired-min-gui.el=: =FEAT/GUI=
  - =emacs-help-gui.el=: =CORE/GUI=
- The boundary report uses this primary group, which removes these 122
  rows from the APP/GUI semantic backlog without moving code.

Extraction rule:

- Command-to-spec policy and writeback ordering may stay here while the
  backend callback boundary remains explicit.
- Any buffer mutation, keymap lookup, minibuffer completion, Dired
  listing semantics, or help text generation that becomes useful outside
  the bridge should move down into =emacs-fileio.el=,
  =emacs-dired-min.el=, =emacs-help.el=, =emacs-buffer-ui.el=, or
  =emacs-minibuffer.el=.

** 2.2 Baked bridge runtime adapter glue

File:

- =src/nemacs-gui-file-bridge-runtime.el= (initially 17, currently 0)

The original matched rows were the =files--minibuffer-gui-backend-*= functions
near the baked runtime image's minibuffer bridge installation.  These callback
symbol names still exist for old standalone images, but the implementations
are installed from =files--minibuffer-gui-backend-operation-table= and the
backend installer is an =fset= adapter, so they no longer appear as individual
baked runtime semantic definitions.

Decision:

- Treat this as compatibility glue for the generated/baked GUI runtime,
  not as the desired long-term owner of minibuffer semantics.
- Keep the public direction established by Doc 20: move reusable runtime
  predicates and minibuffer behavior into shared modules, then leave only
  storage and command-execute wiring here.

Next extraction candidates:

| Candidate | Target owner | Reason |
|-----------+--------------+--------|
| minibuffer text/cursor mutation | CORE via =emacs-minibuffer.el= | reusable prompt editing behavior |
| candidate source selection | CORE/IO adapter boundary | shared command completion surface |
| replace followup state | BUF/FEAT via replace runtime | query-replace should not be bridge-local |
| command-spec execution wrapper | CORE command-loop adapter | command dispatch policy should be shared |

** 2.3 APP/TUI command semantics

File:

- =src/nemacs-main.el= (35)

This group includes:

- keymap construction and installation,
- TUI key dispatch,
- direct printable self-insert fast path,
- interactive file/buffer commands,
- lightweight Dired/Help/Info/Shell/Query-replace commands,
- =M-x= command dispatch.

Decision:

- =nemacs-main.el= should remain the runnable app assembly and event-loop
  owner.
- Reusable semantics inside it should be drained into shared packages in
  small vertical slices.

Priority slices:

| Slice | Current examples | Target owner |
|-------+------------------+--------------|
| TUI key dispatch | =nemacs-main--dispatch-key-code=, =nemacs-main--dispatch-key-event= | CORE command-loop plus DSP/TUI event adapter |
| printable self-insert fast path | =nemacs-main--execute-printable-self-insert= | BUF/edit command runtime |
| file/buffer prompts | =nemacs-main-find-file-interactive=, =nemacs-main-switch-to-buffer-interactive= | IO/BUF command facades with app prompt adapter |
| TUI Dired/Help/Info adapters | =nemacs-main--install-tui-gui-adapters= | FEAT/CORE runtime adapters |
| shell output stub | =nemacs-main-shell-command-interactive= | IO/FEAT shell runtime |
| query replace | =nemacs-main-query-replace-interactive= | BUF/FEAT replace runtime |
| M-x dispatch table | =nemacs-main--run-mx= plus shared command-loop dispatch helpers | CORE command-loop/minibuffer command executor |

Do not move all of this at once.  Each slice should introduce a public
API and a focused test before =nemacs-main.el= is thinned.

** 2.4 Concrete GTK frontend semantics

File:

- =src/nemacs-gtk-frontend.el= (34 reported rows)

The report only sees names matching its semantic regex.  The file contains
more editor behavior than these 34 rows, so the reported count is a lower
bound for extraction work.

Reported clusters:

| Cluster | Examples | Decision |
|---------+----------+----------|
| key event mapping and dispatch | =nemacs-gtk--handle-gdk-input= plus compatibility alias =nemacs-gtk--dispatch-key= | split raw GDK decoding from shared command dispatch |
| minibuffer UI state | =nemacs-gtk--enter-minibuffer=, =nemacs-gtk--minibuffer-handle-key= | move prompt semantics to CORE; keep painting/state bridge in GTK |
| query-replace state machine | =nemacs-gtk-query-replace= and helpers | move replace semantics to BUF/FEAT |
| shell command behavior | =nemacs-gtk-shell-command= and helpers | move shell execution/presentation spec to IO/FEAT |
| help/key description | =nemacs-gtk-describe-key= and helpers | call shared help/keymap APIs |
| file/buffer commands | =nemacs-gtk-keyboard-find-file=, =nemacs-gtk-write-file= | call IO/BUF fileio APIs |
| edit affordances | =nemacs-gtk-undo=, =nemacs-gtk-yank-pop= | call BUF edit/undo/kill-ring APIs |

Decision:

- GTK may own GDK event decoding, painting, cache invalidation,
  clipboard/system integration, and window-system state.
- GTK should not be the long-term owner of command semantics.
- New GTK behavior should first check for a shared =emacs-*= or
  =nelisp-ec-*= API.  If the API does not exist, add it outside GTK.

** 2.5 GUI transport-owned rows

Files:

- =gui/nemacs-editor-transport.el= (4)
- =src/nemacs-gtk-view-menu.el= (1)

The transport rows match words like =command= or =minibuffer=, but most of
this code rewrites transport forms, patches rendering, or wires view/menu
actions.

Decision:

- Keep transport rewriting and concrete menu/view actions GUI-owned.
- Only extract behavior if the code begins deciding editor state or
  command semantics independent of transport.
- Transport patchers should use shape-oriented names, not editor-semantic
  names, so they stay out of semantic backlog counts.  For example,
  =gui/nemacs-editor-transport.el= uses =cmd= and =mb echo area= helper
  names for AOT rewrite shapes instead of =command= / =minibuffer= API names.

  backend callbacks are tracked as TUI frontend glue.
