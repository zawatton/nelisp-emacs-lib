* 0. Purpose

This document defines the public API inventory used during the
library-first extraction phase.

The inventory is generated, not hand-maintained:

#+begin_src sh
make nemacs-public-api-inventory
#+end_src

It reads Doc 18 ownership groups and scans top-level Elisp definitions
under =src/= and =gui/=.  The generated API artifacts are:

- =build/nemacs-public-api-inventory.tsv=
- =build/nemacs-public-api-summary.org=

The narrow external consumer contract is generated separately:

#+begin_src sh
make nemacs-library-contract
#+end_src

It writes:

- =build/nemacs-library-contract.tsv=
- =build/nemacs-library-contract.org=

The contract artifact verifies only the stable facade feature, contract
version variable, and facade query APIs available after requiring
=nelisp-emacs=.  The broader public API inventory remains advisory.

The facade package manifest is generated separately:

#+begin_src sh
make nemacs-library-package-manifest
#+end_src

It writes:

- =build/nemacs-library-package-manifest.tsv=
- =build/nemacs-library-package-manifest-summary.org=

Package dependency artifacts are generated by:

#+begin_src sh
make nemacs-library-package-deps
#+end_src

It writes:

- =build/nemacs-library-package-deps.tsv=
- =build/nemacs-library-package-deps-summary.org=
- =build/nemacs-library-package-migration-queue.tsv=
- =build/nemacs-library-package-migration-queue.org=

The migration queue is derived from =unmanifested-reusable= dependency
edges and groups them by provider feature for package-boundary cleanup.

Draft package descriptor artifacts are generated by:

#+begin_src sh
make nemacs-library-package-descriptors
#+end_src

They write:

- =build/nemacs-library-package-descriptors.tsv=
- =build/nemacs-library-package-descriptors.org=

The descriptor drafts combine facade package membership with generated
dependency edges.  They list source files, facade package dependencies,
lazy features and their source files, host Emacs features, vendor package
features, and unknown external requirements for future package archive or
repository extraction work.

Consumer package guide artifacts are generated by:

#+begin_src sh
make nemacs-library-package-guide
#+end_src

They write:

- =build/nemacs-library-package-guide.tsv=
- =build/nemacs-library-package-guide.org=

The guide is derived from the descriptor drafts and presents the package
loader feature, future package dependencies, lazy companions, and
host/vendor assumptions in a form intended for external consumers.

Package-scoped API artifacts are generated by:

#+begin_src sh
make nemacs-library-package-api
#+end_src

They write:

- =build/nemacs-library-package-api.tsv=
- =build/nemacs-library-package-api.org=

The package API inventory joins descriptor source files with the broader
public API inventory.  It lets package owners inspect candidate
=public-prefixed= APIs, compatibility globals, and private helper surface
by facade package.  It remains advisory and does not by itself make a
symbol stable.

Consumer package API catalog artifacts are generated by:

#+begin_src sh
make nemacs-library-package-catalog
#+end_src

They write:

- =build/nemacs-library-package-catalog.tsv=
- =build/nemacs-library-package-catalog.org=

The catalog joins the package guide with the package API inventory.  It is
the consumer-facing package summary for loader features, package
dependencies, host/vendor assumptions, =public-prefixed= API candidates,
=compat-global= shim names, and private-helper counts.

Package API promotion queue artifacts are generated by:

#+begin_src sh
make nemacs-library-api-promotion-queue
#+end_src

They write:

- =build/nemacs-library-api-promotion-queue.tsv=
- =build/nemacs-library-api-promotion-queue.org=

The promotion queue filters package API rows to =public-prefixed= and
=compat-global= candidates and classifies them by evidence from =test/=,
=README.org=, =AGENTS.md=, =CLAUDE.md=, and =docs/design/=.  The generated
statuses are review inputs, not stability promises:

| Status | Meaning |
|--------+---------|
| =stable-contract= | listed in the facade stable package API manifest or stable lazy package API manifest |
| =promote-ready= | test and doc references exist; package owner can consider documenting it as stable |
| =loader-manifest= | package membership manifest symbol; covered by loader contract |
| =needs-doc= | test reference exists but consumer-facing docs do not |
| =needs-test= | doc reference exists but test coverage does not |
| =needs-review= | no test/doc evidence yet |
| =compat-shim-covered= | compat-global shim has test or doc evidence |
| =compat-shim-review= | compat-global shim needs explicit compatibility review |
| =compat-shim-helper-covered= | prefixed helper in an =emacs-*-builtins.el= shim module has test or doc evidence |
| =compat-shim-helper-internal= | prefixed helper in an =emacs-*-builtins.el= shim module was reviewed and remains internal implementation API |
| =compat-shim-helper-review= | prefixed helper in an =emacs-*-builtins.el= shim module needs explicit compatibility review |
| =validation-helper-covered= | public-prefixed guard helper has test or doc evidence |
| =validation-helper-internal= | public-prefixed guard helper was reviewed and remains internal implementation API |
| =validation-helper-review= | public-prefixed guard helper needs explicit internal API review |

Rows classified as =compat-shim-helper-*= are intentionally not promoted
just because they are prefixed.  They exist to install or back
Emacs-compatible unprefixed names from explicit =*-builtins.el= shim
modules.  External reusable consumers should prefer the facade stable API
or the owning package's non-shim prefixed API instead of depending on
these helper names directly.  =compat-shim-helper-internal= means the
package owner reviewed the helper and decided that it is implementation
surface, not a future stable consumer contract candidate.

Rows classified as =validation-helper-*= are guard/validation helpers,
such as liveness checks.  They are kept out of the stable consumer
contract unless package owners document an external validation use case and
promote them through the facade stable API manifest.
=validation-helper-internal= means the package owner reviewed the helper
and decided that it is implementation surface.

Doc 26 audits the current non-stable residual set after the stable package
API promotion pass.  It records why the remaining =compat-shim-*=,
=loader-manifest=, and =validation-helper-internal= rows are not unfinished
stable API work.

Draft package layout artifacts are generated by:

#+begin_src sh
make nemacs-library-package-layout
#+end_src

They write:

- =build/nemacs-library-package-layout.tsv=
- =build/nemacs-library-package-layout.org=

The layout plan maps descriptor source files to a future
=packages/PACKAGE/lisp/= or =packages/PACKAGE/lazy/= tree.  It is a
review artifact only and does not move files.
Doc 27 audits whether the descriptor, guide, dependency, promotion, and
layout artifacts are ready for a staged physical package extraction.

Package extraction consistency is verified by:

#+begin_src sh
make nemacs-library-package-verify
#+end_src

It writes:

- =build/nemacs-library-package-verify.tsv=
- =build/nemacs-library-package-verify.org=

The verification target checks descriptor uniqueness, source-file
existence, package dependency resolution, unknown external dependencies,
lazy feature/source-file shape, guide/descriptor consistency,
package-scoped API package/file coverage and scope, catalog/API-guide
consistency, promotion queue coverage/status shape, and exact
descriptor/layout coverage.

* 1. Classes

| Class | Meaning | Stability |
|-------+---------+-----------|
| =public-prefixed= | reusable project-prefixed symbols such as =emacs-buffer-*= or =emacs-window-*= | preferred library surface |
| =compat-global= | intentional Emacs-compatible global names such as =find-file=, =dired=, or =describe-function= | compatibility surface |
| =private-helper= | names containing =--= | not public API |
| =adapter-or-app= | APP/GUI-owned definitions | adapter surface, not package API |

The inventory does not promise stability by itself.  A symbol becomes
stable only after the owning package documents and tests it.

* 2. Current baseline

As of 2026-06-22:

| Class | Count |
|-------+-------|
| =public-prefixed= | 1633 |
| =compat-global= | 355 |
| =private-helper= | 2165 |
| =adapter-or-app= | 220 |

All scanned files are owned by Doc 18 groups; there are no =UNOWNED=
rows in the generated summary.

This is a library gate invariant: =make nemacs-library-gate= fails when
=build/nemacs-public-api-inventory.tsv= contains any =UNOWNED= rows.
Doc 18 file-level coverage is checked separately by
=make nemacs-ownership-coverage=; =make nemacs-library-gate= also fails on
coverage =unowned= or =stale= rows.

* 3. Usage rules

- New reusable APIs should be =public-prefixed= unless they intentionally
  emulate an Emacs global API.
- Package group loaders should publish their load membership through
  =*-features= manifests, for example =emacs-buffer-core-features= or
  =emacs-core-features=.
- Hidden reusable substrates used by multiple groups should become either
  explicit package members or explicit facade packages.  =emacs-text-core=
  exposes TXT regex/coding substrates as =text-core=, while
  =nelisp-text-buffer=, =nelisp-emacs-compat=, and =emacs-buffer= are explicit
  =buffer-core= members.
- Editing commands and undo bridges are exposed through the separate
  =editing= package.  =core= may depend on =editing=, but should not claim
  BUF editing members as part of its own package membership.
- Eager package loaders should name all mandatory same-owner substrates in
  their =*-features= manifests.  Nested/lazy =require= forms may be
  declared as package lazy companions.  They are reported as
  =lazy-manifest-package= and should not be promoted to eager membership
  merely to make optional data or frontend hooks load.  Undeclared lazy
  reusable edges stay visible as =lazy-unmanifested-reusable= for review.
- The top-level facade should expose the package-group manifest through
  =nelisp-emacs-library-packages= and
  =nelisp-emacs-library-package-manifest= so external consumers can inspect
  reusable package membership programmatically.
- Consumers that need a narrower view should use
  =nelisp-emacs-library-package-names=,
  =nelisp-emacs-library-package=,
  =nelisp-emacs-library-package-features=, and
  =nelisp-emacs-library-package-lazy-features= instead of depending on the
  raw manifest shape.
- The external consumer contract is versioned separately by
  =nelisp-emacs-library-contract-version= and specified in Doc 25.  The
  inventory remains broader than that contract.
- Use =make nemacs-library-contract= when changing facade query APIs or
  the external consumer contract version.
- Use =make nemacs-library-package-manifest= when consumers or package
  extraction work need a generated owner/feature/member/lazy view of the
  facade.
- Use =make nemacs-library-package-deps= to review generated package group
  dependency edges.  =make nemacs-library-gate= fails when the dependency
  report contains =app-or-frontend=, =unmanifested-reusable=,
  =lazy-unmanifested-reusable=, or =external-or-host= rows.  Use the
  generated migration queue to reduce or intentionally document eager
  =unmanifested-reusable= rows before they reach the gate.
- Use =make nemacs-library-package-descriptors= when package archive,
  package.el, ELPA-style, or repository extraction work needs a generated
  package descriptor draft.
- Use =make nemacs-library-package-guide= when external-consumer docs need
  the generated package loader/dependency/lazy-feature view.
- Use =make nemacs-library-package-api= when package owners or consumers
  need a package-scoped candidate API view.
- Use =make nemacs-library-package-catalog= when external consumers need a
  package-oriented loader/dependency/API summary derived from generated
  guide and API artifacts.
- Use =make nemacs-library-api-promotion-queue= when package API surface,
  package docs, or package tests change; it identifies which candidates
  need docs, tests, or owner review before they become consumer-facing API.
- Add eager symbols to =nelisp-emacs-library-stable-package-api= only
  after the promotion queue shows sufficient test/doc evidence and
  package-owner review.  Add explicit-load symbols to
  =nelisp-emacs-library-stable-lazy-package-api= only when their lazy
  feature is documented and verified.  Stable symbols must classify as
  =stable-contract= under =make nemacs-library-api-promotion-queue=.
- Use =make nemacs-library-package-layout= when physical package
  extraction work needs a generated =packages/= move plan.
- Use =make nemacs-library-package-verify= before relying on descriptor,
  guide, package API, catalog, promotion queue, or layout artifacts for
  package extraction work.
- External-consumer compatibility is checked by
  =make nemacs-library-consumer-smoke=.  The target loads =nelisp-emacs=
  from =emacs -Q -L src= and must not load app/bootstrap, concrete
  TUI/GUI, or lazy editor files.
- Package group extraction compatibility is checked by
  =make nemacs-library-package-smoke=.  The target requires each reusable
  package group loader in its own =emacs -Q -L src= process.
- Reusable feature manifests must stay non-empty, duplicate-free,
  resolvable through =locate-library=, and free of app/bootstrap,
  concrete TUI/GUI, and lazy editor features.
- GUI, TUI, and app entrypoints should consume =public-prefixed= or
  =compat-global= APIs instead of calling =private-helper= names across
  ownership groups.
- Before extracting a group into a package, review that group's rows in
  =build/nemacs-public-api-inventory.tsv= and choose which symbols are
  documented stable API.
- If the inventory reports =UNOWNED=, update Doc 18 ownership before
  treating the new file as part of the reusable library surface.
- =compat-global= rows are valuable for Emacs compatibility, but external
  projects should prefer the prefixed API when both exist.
