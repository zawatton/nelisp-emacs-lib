* §0 motivation

** 0.1 user directive (2026-05-17)

"vendor/ 下にある Emacs ソースコードの elisp が動作するように出来れば完成"
+ "nelisp 側に主戦場を移すにしても、実装は Rust ではなく elisp で実装
(Rust コア削除)"

= 「Emacs 置き換え完遂」の user 定義は *vendor/emacs-lisp/ 配下 1440 el が
NeLisp ランタイム上で動く* こと。Phase 4 (= MELPA scratch compat) や
Phase 2 (= TUI MVP scratch) は user 目標と方向違い (= scratch 実装増殖)。

** 0.2 v0.2 vendor migration v1 revert (2026-05-09) からの学習

a9c2f58 / 0aa3b24 で help-fns / help-mode / lisp-mode / ielm / minibuffer /
project / files / simple / isearch / dired の 11 module を vendor 経由で
動作確認 (= 27 commands fboundp OK + 6 runtime PASS) → 0.2 M3 全 5 module
(org-mode / cus-theme / babel / export) 含めて 16 module vendor 化。

VMware Debian での verification で破綻 → 95316e2 で revert。真因は
NeLisp Rust runtime の 4 つの limitation:

1. *reader* — =?'= (apostrophe char literal) 直接受け入れ不可、
   "lex error at 9041:49: empty char literal after =?=" を吐く。
   vendor files.el / info.el / woman.el 等で頻出
2. *function-quote* — =#'macro= 受け入れ不可
3. *autoload-defs* — dump 相当の pre-load なし、=autoload= 経由で
   呼ばれる関数が見つからない
4. *dump* — preloaded.el / image bake 機構なし、global state を image
   として焼き付けて起動高速化する path がない

これら 4 件は *NeLisp Rust runtime に深い限界* で、vendor 全体を render
可能にするには NeLisp 改善必須。1-char workaround (39ff46f: =?'= → =?\'=)
で vendor pristine が崩れた事も revert 理由。

** 0.3 本 doc の主張

1. v0.2 migration v1 で名指しされた 4 blocker は *既に dev/nelisp/ 側で
   pure-elisp self-host migration 道筋 (Doc 83) に組み込まれている*
   → blocker 解消は dev/nelisp/ 側の Doc 73/75/77c/102/104 等の SHIPPED
   timeline と完全に同期する
2. 1440 vendor el は均一ではなく、blocker 依存度で *5 段階クラス分け* 可能
   → クラス A (現 NeLisp で既に動く ~11 module = v0.2 verified) から
   段階的に vendor 起動できる
3. v0.1 scratch (= 17 module) との *共存戦略* = vendor を上に top する、
   scratch を破棄しない。v0.2 v1 では "redundant" として削除して revert
   時に LOC +7439 復活で失敗。v2 では vendor が動いた module のみ scratch を
   優先後退 (= require 順 vendor → scratch fallback)
4. *起動 verification methodology* = guilemacs / rune の先行事例から学習、
   ERT mirror + property comparison + loadup.el bootstrap + smoke 関数の
   4 軸で進捗計測。これが無いと「動いた」の定義が曖昧

** 2.1 guilemacs (codeberg.org/lyrra/guilemacs)

Guile Scheme runtime 上で Elisp engine を載せる exploratory fork。
test methodology:
- *ERT mirror*: 本家 Emacs ERT 1741 cases をそのまま走らせて pass/fail
  集計 (= 多くは fail だが指標として有効)
- *Prelude tests*: C 定義 primitive の自前 unit test (=test/pre/=)、
  loadup.el 前の core primitive 検証

  target binary に load → execute
- *Bootstrap*: =loadup.el= 完走を critical gate に設定
- *Smoke 関数*: Tetris / Snake / Dunnet / Eshell が起動するか

180 generated tests + 1741 ERT で recent build = all pass を維持。

** 2.2 rune (github.com/CeleritasCelery/rune)

Rust runtime 上で Elisp VM を実装する exploratory project。test methodology:
- *property comparison*: =assert_elprop!= macro で同じ form を rune と
  host Emacs に送り、結果を比較。互換性検証の主軸
- *exploratory*: "try to load new elisp files and see what breaks"
- vendor lisp = emacs-mirror emacs-29.1 branch を pull、=/lisp/= 配下に配置
- vendor patch は =RUNE-BOOTSTRAP= tag で source 内 isolate
- =cargo run --release -- --load <file>= で個別 file load 駆動

* §4 blocker taxonomy + dev/nelisp/ mapping

** 4.1 blocker #1: reader =?'=

- 症状: =vendor/emacs-lisp/files.el:9041:49=、
  =info.el:4485=、=woman.el:...= 等で "lex error after =?=" 多発
- 短期 workaround: =?'= → =?\'= の 1-char patch (v0.2 v1 で 39ff46f
  実施、revert)。pristine 破壊で reject
- 真因解消: NeLisp Rust reader に =?'= = ASCII 39 apostrophe char literal
  を直接受け入れる branch 追加
- dev/nelisp/ mapping: *Doc 73 Stage 8.4 reader cfg-gate* (-1,679 Rust LOC、
  Track A Phase B5 SHIPPED 後 dispatch unblock)。Doc 83 §3.3 milestone
  table の "Doc 73 Stage 8.4" 行が gate

** 4.2 blocker #2: function-quote =#'macro=

- 症状: =#'macro-name= の形で macro symbol を function-position に
  bind する慣用句が vendor easy-mmode.el line 447 等で使われる、
  v0.2 v1 で 5e15b63 + 484a0ae の 2 site patch (revert)
- 真因解消: NeLisp Rust reader + eval が =#'= prefix を quote の variant
  として handle、macro symbol も function-quotable に
- dev/nelisp/ mapping: *Doc 73 Phase 8 rust-reader-eval-purge* および
  *Doc 77c sexp-abi-elisp-self-host*。Doc 67 phase7-stage3 special-forms
  migration 配下の =function= form / =#'= reader entry も関係

** 4.3 blocker #3: autoload-defs (= dump 相当の pre-load)

- 症状: vendor の =autoload= 経由 function が "void-function" になる。
  =cus-load.el= / =finder-inf.el= / =loaddefs.el= 等の pre-baked 巨大
  =autoload= 宣言群を起動前に inject する path が NeLisp に無い
- 真因解消: =preloaded.el= / =frozen heap image= 機構の整備
- dev/nelisp/ mapping: *Doc 75 phase9 frozen heap* (-1,524 net Rust)
  + *Doc 28 §3.7 autoload* + *Doc 81 image-baker*

** 4.4 blocker #4: dump (= preloaded.el 機構)

- 症状: GNU Emacs は =build_emacs= 時に =loadup.el= で大量 .el を
  load → =dump-emacs-portable= で =emacs.pdmp= に焼く。NeLisp は dump
  step 不在 = 毎起動 loadup を回す path 必要
- 真因解消: image baker + frozen heap で同等の "pre-loaded state を
  binary に焼く" 機構を pure-elisp で実装
- dev/nelisp/ mapping: *Doc 81 image-baker* + *Doc 75 phase9 frozen heap*
  + *Doc 98 stage-95-real-frozen-heap-baker*

** 4.5 まとめ: dev/nelisp/ SHIPPED → vendor unlock 関係

| dev/nelisp/ ship | unlocks vendor class | vendor el 増分見込 |
|------------------+----------------------+--------------------|
| Doc 73 Stage 8.4 reader cfg-gate | A → B | +100-200 module |

* §6 起動 verification suite design

** 6.1 4 軸構成

| 軸 | 目的 | 出力 | 採用元 |
|----+------+------+--------|
| A. ERT mirror | 本家 Emacs ERT case 走行 → pass/fail/skip 集計 | =test/ert-mirror-status.csv= | guilemacs |
| B. property comparison | 同 form を nelisp-emacs と host Emacs に送り結果比較 | =test/elprop-results.csv= | rune |
| C. loadup bootstrap | =loadup.el= chain を上から walk、どこで break するか追跡 | =test/loadup-walk.log= | guilemacs + rune |
| D. smoke functional | Tetris / Snake / Dunnet / Eshell / IRC が起動 + 1 操作完走 | =test/smoke-functional.log= | guilemacs |

** 6.2 軸 A: ERT mirror

スコープ (= 10.3 確定):
- *base*: daily-driver 関連 ~50-100 file を =vendor/emacs-tests/= に
  curated vendor (= files-tests.el / simple-tests.el / isearch-tests.el /
  dired-tests.el / org/*-tests.el / help-fns-tests.el / etc.)
- *spot full check*: 要所 (= phase close / major release 取り込み後 /
  weekly nightly) で host Emacs =test/lisp/= 全件 (~数千 case) を走らせ
  pass% を計測

実装:
- nelisp-emacs binary で load → run → 結果集計
- skip 条件 (= NeLisp blocker 既知) を =skip-when-blocker= macro で分類
- 出力: csv で {file, test, status, blocker-class}
- progress KPI (base scope): pass% (= goal Phase 1: ≥10%, Phase 2: ≥30%,
  Phase 3: ≥50%, Phase 4: ≥80%, Phase 5: ≥95%)
- progress KPI (full scope spot check): regression detection (= pass%
  decrease > 5% で CI alert)

** 6.3 軸 B: property comparison (= rune =assert_elprop!= 風)

form 集生成 (= 10.4 確定): =scripts/extract-elprop-forms.el= で host Emacs
=test/lisp/*-tests.el= の =should= form を auto-extract、pure form のみを
filter。Phase 1 で ~500 form 生成、Phase 2+ で拡張。

実装:
- =test/elprop-forms-auto.el= に (id, form, expected, type) リスト
- =bin/elprop-run= が:
  1. host Emacs (=emacs --batch=) に form を投入 → 結果取得
  2. nelisp-emacs binary に同 form 投入 → 結果取得
  3. =equal= で比較、不一致なら diff 出力
- form 例 (= 抽出元 ERT body): =(+ 1 2)=, =(mapcar #'1+ '(1 2 3))=,
  =(let ((x 10)) (* x x))=, =(substring "hello" 1 3)=, ...
- 出力: csv で {form-id, host-result, nelisp-result, match?}

** 6.4 軸 C: loadup bootstrap walk

- =vendor/emacs-lisp/loadup.el= を実 load せず static parse → file 依存
  graph build
- =bin/loadup-walker= が graph を上から順に =(load file)= 試行、
  initial error の file 名 + error 種別を log
- 出力: =loadup-walk.log= で "file X で symbol Y not bound" 等を抽出
- progress KPI: chain 完走 file 数 (= goal Phase 1: ~50, Phase 5: 全)

** 6.5 軸 D: smoke functional

- 各 smoke は subprocess で =nelisp-emacs -nw -l <file> -f <fn>= 起動、
  expect 出力を timeout 内に検出
- 候補 (= guilemacs 同):
  1. =tetris= = =M-x tetris= 起動 + 任意 key 1 操作 + quit
  2. =snake= = 同様
  3. =dunnet= = =M-x dunnet= 起動 + =go east= 1 step
  4. =eshell= = =M-x eshell= 起動 + =ls= 実行
  5. =org-agenda= = =org-agenda= 起動 + view dispatch
- 出力: 各 smoke の pass/fail + 所要 ms

** 6.6 CI integration

- =Makefile= に =make verify-vendor= target、4 軸を順次 run
- GitHub Actions で nightly + main push 時に走行
- regression detection = pass% 減少 / smoke fail を CI で alert

* §7 v0.1 scratch との共存戦略

** 7.1 require 順序 規約

vendor を *先*、scratch を *fallback*:

#+begin_src elisp
;; nemacs-gtk-frontend.el / init.el 等の中で:
(condition-case err
    (progn
      (require 'files)            ; vendor files.el
      (require 'simple)           ; vendor simple.el
      (require 'dired))           ; vendor dired.el
  (error
    (message "vendor load failed: %s. fallback to scratch." err)
    (require 'emacs-fileio)        ; v0.1 scratch
    (require 'emacs-buffer-ui)
    (require 'emacs-dired-min)))
#+end_src

** 7.2 fboundp 条件付き scratch defun

scratch 側は vendor 同名関数を上書きしない:

#+begin_src elisp
;; emacs-fileio.el の中で:
(unless (fboundp 'find-file)
  (defun find-file (filename) ...))
#+end_src

= 既存 v0.1 scratch を全件この pattern に修正 (= =feedback_emacs_stub_
unconditional_defun_clobber= の rule 適用拡張)

** 7.3 scratch removal gate

class A vendor 化 + 1 ヶ月 soak で regression 0 確認後、scratch 削除を許可:
- 個別 module 単位で remove (= 1 commit = 1 module)
- 削除前 1 週間は =message= で deprecation 警告
- 削除コミットは ERT + smoke を緑で確認

** 7.4 vendor patch tag (= =NELISP-EMACS-BOOTSTRAP=)

vendor pristine が原則だが、blocker 解消前に短期 patch が必要な場合:

** 10.1 vendor storage: git subtree (squash flag)

vendor/emacs-lisp/ = emacs-mirror emacs-30 branch の git subtree。半年毎
major upgrade 時の workflow:

#+begin_src bash
git subtree pull \
  --prefix=vendor/emacs-lisp \
  https://github.com/emacs-mirror/emacs.git \
  emacs-30 \
  --squash \
  -m "vendor: bump to emacs 30.x"
#+end_src

- squash flag で git history 上 1 commit / 1 update に圧縮 (= noise 削減)
- patch は inline commit (= NELISP-EMACS-BOOTSTRAP tag で audit 可能、§7.4)
- emacs-31 リリース時は =--prefix=vendor/emacs-lisp emacs-31= に切り替え
- rune と同じ pull-from-emacs-mirror pattern

scripts/refresh-vendor.sh = subtree pull の wrapper、conflict 時に
patch reverify を半自動化。

** 10.2 host Emacs version pin: 半年毎 major 上げ

- 起点: 現在 vendor 取り込み済の Emacs 30.x (= 2026-05-17 baseline)
- update cadence: ~半年毎、major release (30.1 → 30.2 → 31.x) に追従
- minor patch (= 30.1.1 → 30.1.2 等) は必要時のみ (= bug fix が
  vendor migration を blocker 化した場合)
- update 後 always re-run §5.1 audit script + §6 verification suite で
  regression detection

** 10.3 ERT mirror スコープ: daily-driver 範囲 file 選択 (base) + 要所で全件

- *base scope* (= Phase 1-4 各 close gate): host Emacs =test/lisp/= から
  daily-driver 関連 file のみ import (~50-100 file)
  - 例: =files-tests.el= / =simple-tests.el= / =isearch-tests.el= /
    =dired-tests.el= / =org-tests.el= / =help-tests.el= / etc.
- *spot full check*: 要所 (= phase ship 直前、major release 取り込み後、
  四半期毎 nightly) で host Emacs =test/lisp/= 全件 (~数千 case) を走らせ
  て pass% を計測、recess 検出
- CI 設定: daily run = base scope、weekly nightly = full

** 10.4 property comparison form 集: host Emacs ERT の =should= 抽出

scripts/extract-elprop-forms.el で host Emacs =test/lisp/*-tests.el= を
parse:

#+begin_src elisp
(defun extract-elprop-forms (file)
  "Extract pure should-forms from host ERT test FILE."
  ;; (should EXPR) → EXPR
  ;; (should-equal A B) → (list :form A :expected B)
  ;; (should-error EXPR :type 'TYPE) → (list :form EXPR :error TYPE)
  ;; pure filter: no setq/save-excursion/with-temp-buffer/...
  ...)
#+end_src

- pure form (= side-effect なし、buffer 状態に依存しない) のみを抽出
- 出力 = =test/elprop-forms-auto.el= の (id, form, expected, type) リスト
- ~500 form を Phase 1 で生成、Phase 2+ で再生成して拡張
