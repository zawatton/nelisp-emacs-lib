#+TITLE: standalone-save-and-transform 次段開発指示書
#+DATE: 2026-06-20

* 目的

=standalone-save-and-transform= の実行時間を大幅に短縮する。現在の主因は
GUI file bridge の host/runner 往復だけではなく、=nemacs-gui-file-bridge-run=
内部の read/writeback dispatch cost が 336 request に広く分散していることが
実測で分かった。単純な wait 調整や小さい request batch では十分な効果が出ない。

次段では、runtime state reset 境界を明確にしたうえで、意味単位ごとに
case を分割・集約する。

* 現状

- persistent runner は導入済み。
- host 側 poll interval は 1ms まで短縮済み。
- publish settle は 1ms だと flaky になるため 5ms を維持する。
- =standalone-save-and-transform= は安全設定で約 108 秒。
- 13 request 程度の小 batch 化は試したが、5ms publish settle では短縮効果がなく不採用。
- motion group 6 request の result file batch 化も試したが、108.432 秒で短縮しなかったため不採用。
- =NEMACS_GUI_BRIDGE_PROFILE=1= での baseline は 336 request、profile 合計 107.901 秒。
  - 1 秒以上の request は 14 件。
  - 遅延は一部の host publish ではなく、多数の bridge run に分散している。
- runtime writeback fast path も試したが、=pop-global-mark= 系の state writeback を落として
  =/tmp/nemacs-global-mark= missing になり、140 秒台へ悪化したため不採用。
- writeback dispatch table 化も試したが、broad table は 124.498 秒へ悪化し、
  motion command 限定 table でも 108.649 秒で 5 秒以上の短縮に届かなかったため不採用。
- =nemacs-gui-file-bridge-run= wrapper 型の motion minimal read path も試したが、
  117.294 秒へ悪化したため不採用。
- =standalone-session-bridge-roundtrip= と =make test-fast= は green。

* 守ること

- publish settle を 1ms に戻して短縮したことにしない。
  - =/tmp/nemacs-file= missing が発生し、fresh process/profile 前提で flaky になる。
- host 側で session/progn を単純に一括化しない。
  - buffer、tab、visited file、modified flag、temp file state が持ち越されて誤判定しやすい。
- 小さい request を数個まとめるだけの batch 化を優先しない。
  - transport complexity が増える割に、支配的な wait を削れない。
- 失敗時にどの subcase が壊れたか分からない巨大 batch にしない。
- command ごとの writeback state を理解せずに fast path を追加しない。
  - =files--write-transport-point= / mark だけで足りる command と、
    global mark、goal column、rectangle、kill ring、window/tab/frame state も必要な command が混在する。

* 開発方針

1. =standalone-save-and-transform= 内の subcase と profile 上位 command を一覧化する。
2. 各 subcase を state/writeback 依存で分類する。
   - 独立実行可能: reset 後に同じ初期条件から動くもの。
   - buffer/file 依存: visited file、modified flag、save/write の順序に意味があるもの。
   - transport 依存: request/response timing や publish order に敏感なもの。
   - writeback 依存: point/mark だけでなく、global mark、goal column、kill ring、
     window/tab/frame、modeline、status file などの副作用を持つもの。
3. =nemacs-test-runner-reset-runtime-state= の reset 範囲を確認し、不足があれば明示的 helper を追加する。
4. 小さい runner batch や runtime の writeback dispatch table ではなく、
   transport input read の最小化、または writeback chain の生成元変更を検討する。
5. command ごとの副作用 boundary を保ったまま、read/writeback のどちらが支配的かを
   小さい group で計測する。
6. 5 秒以上の短縮がない group は戻し、doc に不採用理由を残す。

* 不採用 group

以下は試行済みで、採用基準に届かなかった。

** motion group

- =forward-char=
- =backward-char=
- =next-line=
- =previous-line=
- =beginning-of-line=
- =end-of-line=

結果:

- result file 方式で 6 request を 1 request にまとめても 108.432 秒。
- writeback dispatch table 方式でも 108.649 秒。
- baseline 108 秒台から 5 秒以上短縮できなかった。
- 採用しない。

** writeback dispatch table

- broad table: point-only / point+mark / buffer 系などをまとめて table hit させる案。
- motion 限定 table: =forward-char= / =backward-char= / =next-line= /
  =previous-line= / =beginning-of-line= / =end-of-line= だけを対象にする案。

結果:

- broad table は =standalone-save-and-transform= 124.498 秒で悪化。
- motion 限定 table は 108.649 秒で baseline 108 秒台から有意に短縮しなかった。
- table lookup helper と既存 chain fallback の併用では、dispatch cost を十分削れない。
- 次は table 追加ではなく、transport input read の無駄削減か、
  command-specific writeback chain の生成元を直接変える。

** wrapper 型 minimal read path

- 既存の巨大 =nemacs-gui-file-bridge-run= を =files--gui-file-bridge-run-full= として残し、
  wrapper で direct motion command だけ minimal read path へ分岐する案。
- 対象は =forward-char= / =backward-char= / =next-line= / =previous-line= /
  =beginning-of-line= / =end-of-line=。
- minimal read には cmd/keys、arg/minibuffer/prefix、buffer、buffer-name、read-only、
  window layout/selected/start、point/mark、goal-column、narrow/minibuffer state を含めた。

結果:

- =standalone-save-and-transform= は pass したが 117.294 秒で悪化。
- miss path でも wrapper の cmd read が追加され、hit path も command-loop/writeback policy の
  再構成 cost が重く、十分な効果が出なかった。
- runtime への後付け wrapper ではなく、生成元または full read body 自体を整理する必要がある。

** local editing group

- =self-insert-command=
- =newline=
- =delete-char=
- =delete-backward-char=
- =delete-trailing-whitespace=

** transform group

- buffer 内容を固定入力から作り直せる変換系

save/write/visited-file lifecycle に触る case は後回しにする。

上記 local editing / transform も、単純な runner batch や table fallback 追加では
効果が薄い可能性が高い。次にやる場合は transport read/write の支配箇所を分けてから
小さい group で検証する。

* result file 形式

runner 側で temp file に sexp list を書く。host 側はその list を読み、subcase ごとに
=ok=、=value=、=error= を検証する。

例:

#+begin_src elisp
((motion-forward-char . (:ok t :point 2 :buffer "abc"))
 (motion-next-line . (:ok t :point 6 :buffer "a\nbc"))
 (editing-delete-char . (:ok t :buffer "ac"))
 (editing-newline . (:ok nil :error "expected point mismatch")))
#+end_src

最低限、各 subcase には次を含める。

- stable name
- ok flag
- expected/actual の比較に必要な値
- error message

* 次の実装単位

次の PR/commit は後付け wrapper/table 追加ではなく、=nemacs-gui-file-bridge-run= 本体または
生成元の read/writeback chain を整理する。

目標:

- 既存の command-specific writeback chain は一気に置換しない。
- command 実行前に全 transport file を読む現状から、command group ごとの read set を
  本体構造として切り出す。
- まず motion command だけを対象にし、buffer / point / mark / window-start など必要 state を明示する。
- kill ring、rectangle、global mark、tab/window/frame、coding/input、custom state などを
  必要とする command は初回対象にしない。
- miss 時の fallback で余分な transport read を増やさない。

後付け wrapper 型 minimal read path は 117.294 秒で悪化したため、同じ形では続けない。
次は runtime 生成元で read/writeback chain 全体を整理する。

* 採用基準

- =standalone-save-and-transform= が安定して短縮する。
- publish settle 5ms のまま通る。
- fresh process 前提の検証で通る。
- =standalone-session-bridge-roundtrip= が通る。
- =make test-fast= が通る。
- 失敗時の subcase 名と actual value が読める。
- read/writeback chain 整理では、未登録 command の挙動が変わらない。

* 次の作業開始点

1. runtime 生成元がないか再確認し、なければ本体内で full read/writeback body を
   機械的に helper 化する。
2. helper 化後、miss path の追加 read が 0 であることを確認する。
3. command group ごとに read set / writeback set を同じ helper 境界へ移す。
4. motion group だけでなく、profile 上位 request の command 分布を先に集計する。
5. =standalone-save-and-transform= と =standalone-session-bridge-roundtrip= で計測する。

小さい request batch、table fallback、後付け wrapper ではなく、command ごとの
read/writeback 副作用を保持した本体構造の整理を先に作ること。
