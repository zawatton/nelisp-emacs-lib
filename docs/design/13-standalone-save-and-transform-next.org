#+TITLE: standalone-save-and-transform 次段開発指示書
#+DATE: 2026-06-20

* 目的

=standalone-save-and-transform= の実行時間を大幅に短縮する。現在の主因は
GUI file bridge の host/runner 往復だけではなく、=nemacs-gui-file-bridge-run=
内部の read/writeback dispatch cost が 336 request に広く分散していることが
実測で分かった。単純な wait 調整や小さい request batch では十分な効果が出ない。

次段では、runtime state reset 境界を明確にしたうえで、意味単位ごとに
case を分割・集約する。

* 現状

- persistent runner は導入済み。
- host 側 poll interval は 1ms まで短縮済み。
- publish settle は 1ms だと flaky になるため 5ms を維持する。
- =standalone-save-and-transform= は安全設定で約 108 秒。
- 13 request 程度の小 batch 化は試したが、5ms publish settle では短縮効果がなく不採用。
- motion group 6 request の result file batch 化も試したが、108.432 秒で短縮しなかったため不採用。
- =NEMACS_GUI_BRIDGE_PROFILE=1= での baseline は 336 request、profile 合計 107.901 秒。
  - 1 秒以上の request は 14 件。
  - 遅延は一部の host publish ではなく、多数の bridge run に分散している。
- runtime writeback fast path も試したが、=pop-global-mark= 系の state writeback を落として
  =/tmp/nemacs-global-mark= missing になり、140 秒台へ悪化したため不採用。
- =standalone-session-bridge-roundtrip= と =make test-fast= は green。

* 守ること

- publish settle を 1ms に戻して短縮したことにしない。
  - =/tmp/nemacs-file= missing が発生し、fresh process/profile 前提で flaky になる。
- host 側で session/progn を単純に一括化しない。
  - buffer、tab、visited file、modified flag、temp file state が持ち越されて誤判定しやすい。
- 小さい request を数個まとめるだけの batch 化を優先しない。
  - transport complexity が増える割に、支配的な wait を削れない。
- 失敗時にどの subcase が壊れたか分からない巨大 batch にしない。
- command ごとの writeback state を理解せずに fast path を追加しない。
  - =files--write-transport-point= / mark だけで足りる command と、
    global mark、goal column、rectangle、kill ring、window/tab/frame state も必要な command が混在する。

* 開発方針

1. =standalone-save-and-transform= 内の subcase と profile 上位 command を一覧化する。
2. 各 subcase を state/writeback 依存で分類する。
   - 独立実行可能: reset 後に同じ初期条件から動くもの。
   - buffer/file 依存: visited file、modified flag、save/write の順序に意味があるもの。
   - transport 依存: request/response timing や publish order に敏感なもの。
   - writeback 依存: point/mark だけでなく、global mark、goal column、kill ring、
     window/tab/frame、modeline、status file などの副作用を持つもの。
3. =nemacs-test-runner-reset-runtime-state= の reset 範囲を確認し、不足があれば明示的 helper を追加する。
4. 小さい runner batch ではなく、runtime の writeback dispatch を生成単位で改善する。
5. command → writeback spec の table を作り、既存 chain と同じ副作用を table から実行する。
6. table 化した command group ごとに計測し、短縮効果がない group は戻す。

* 不採用 group

以下は試行済みで、採用基準に届かなかった。

** motion group

- =forward-char=
- =backward-char=
- =next-line=
- =previous-line=
- =beginning-of-line=
- =end-of-line=

結果:

- result file 方式で 6 request を 1 request にまとめても 108.432 秒。
- baseline 108 秒台から 5 秒以上短縮できなかった。
- 採用しない。

** local editing group

- =self-insert-command=
- =newline=
- =delete-char=
- =delete-backward-char=
- =delete-trailing-whitespace=

** transform group

- buffer 内容を固定入力から作り直せる変換系

save/write/visited-file lifecycle に触る case は後回しにする。

上記 local editing / transform も、単純な runner batch では motion group と同じ理由で
効果が薄い可能性が高い。次にやる場合は batch 化ではなく writeback dispatch table 化と
組み合わせる。

* result file 形式

runner 側で temp file に sexp list を書く。host 側はその list を読み、subcase ごとに
=ok=、=value=、=error= を検証する。

例:

#+begin_src elisp
((motion-forward-char . (:ok t :point 2 :buffer "abc"))
 (motion-next-line . (:ok t :point 6 :buffer "a\nbc"))
 (editing-delete-char . (:ok t :buffer "ac"))
 (editing-newline . (:ok nil :error "expected point mismatch")))
#+end_src

最低限、各 subcase には次を含める。

- stable name
- ok flag
- expected/actual の比較に必要な値
- error message

* 次の実装単位

最初の PR/commit は writeback dispatch の table 化にする。

目標:

- 既存の command-specific writeback chain を一気に置換しない。
- point-only / point+mark / buffer+point / buffer+point+mark のような副作用 spec を定義する。
- まず motion command だけを table 経由にする。
- table 未登録 command は既存 chain に fallback する。
- =pop-global-mark= など追加 state を持つ command は初回対象にしない。

motion command の table 化で 5 秒以上短縮できない場合は、単なる dispatch table だけでは
不十分と判断し、writeback chain 全体の生成元を変更する。

* 採用基準

- =standalone-save-and-transform= が安定して短縮する。
- publish settle 5ms のまま通る。
- fresh process 前提の検証で通る。
- =standalone-session-bridge-roundtrip= が通る。
- =make test-fast= が通る。
- 失敗時の subcase 名と actual value が読める。
- writeback table 化では、既存 chain fallback が残り、未登録 command の挙動が変わらない。

* 次の作業開始点

1. =test/nemacs-gui-file-bridge-runtime-test.el= の
   profile 上位 command と writeback assertion を読む。
2. =src/nemacs-gui-file-bridge-runtime.el= の writeback chain から
   point-only command の spec を抜き出す。
3. =files--writeback-spec-current-context= のような table lookup helper を追加する。
4. table hit 時だけ spec writeback を実行し、miss は既存 chain に fallback する。
5. motion command だけ table 化して計測する。

小さい request batch ではなく、command ごとの writeback 副作用を保持した dispatch 改善を
先に作ること。
