* 0. Purpose / 位置づけ

Doc 11 / Doc 14 が「何が残っているか」を列挙した roadmap なのに対し、本書は
user 指示 (2026-06-20) による *実行順序の確定* — まず小物を片付け、次に中物、
最後に大物、という波で進める。各波の中の項目は独立着手可能 (= parallel
agent 向き) に保つ。

直近の *OS 判別* (=uname(2)= runtime detection, =src/emacs-os-detect.el=,
commit f0ba42b) が「小物波」の最初のドミノであり、以降の小物が踏襲すべき
*型* でもある:

- 純 Elisp、Rust LOC 不変 (nelisp-emacs は reader substrate を *消費する側*)。
- 既存 substrate (=syscall-direct= / =alloc-bytes= / =getenv= 等) に乗せる。
- host Emacs driver では no-op、reader でのみ作用 (=fboundp= gate)。
- ハードコード default は残し、検出/解決できた時だけ上書き (= 安全な degrade)。
- ERT は host 純関数 + standalone reader gate の 2 レーン。
- canonical に =.elc= を残さない / =make build-nelisp-bootstrap= で bundle 再生成。

** 2.2 実装メモ (2026-06-20, A1/A2 着地)

- *A1 の真因*: =getenv= は fboundp だったが、reader prelude が空の
  =nelisp--environment= を読む getenv を先に定義しており、=emacs-callproc.el=
  の =(unless (fboundp 'getenv) …)= ガードが正しい =process-environment= 版を
  install できずに shadowing されていた (=process-environment= 自体は
  =/proc/self/environ= から seed 済で HOME 等を保持)。
- *fix*: =emacs-callproc.el= の getenv/setenv ガードを
  =(when (or (not (fboundp 'getenv)) (fboundp 'rdf)) …)= に変更。reader
  (=rdf= 在) では =process-environment= 版を強制 install、host (=rdf= 不在) では
  C builtin を保護。これが getenv 系全体 (executable-find の PATH walk 含む) の
  foundation。
- *A1/A2 本体*: =src/emacs-os-detect.el= に =emacs-os--env= /
  =emacs-os-apply-os-polyfills!= / =emacs-os-detect-and-set-dirs!= を追加、
  load-time apply を system-type → path-separator → PATH-split の順で実行。
- *検証*: full boot で =user-emacs-directory= = HOME 由来、=exec-path= = 実 PATH
  (14 要素)、=path-separator= ":"。host ERT 14/14 (standalone reader gate 込)。

** 3.2 B2 診断訂正 (2026-06-20, 着手調査)

Doc 14 (2026-06-14) は「bridge runtime に file-read/buffer ops 無し (fboundp
nil)」と記したが、その後の bridge image 再 bake で状況が変わっており、実地
調査で *診断を訂正*:

- *file-read/buffer-ops は実装済で full =--batch= stack では完動*:
  =insert-file-contents-literally= は BEG/END range を honor
  (=nelisp-ec-insert-file-contents= = =nelisp-emacs-compat-fileio.el:625=、
  =nelisp-ec--read-raw-bytes file beg end= + =nelisp-ec-insert=)、
  =buffer-substring=/=-no-properties= は nelisp-ec 本実装、=point-min/max= 実動。
  cdb read pattern が byte-exact に =(65 66 67 230)= を返す (binary 高 byte 保持)。
- *真の gap = GUI bridge image (=build/nemacs-gui-file-bridge.nlri=)*:
  そこでは全 op が fboundp t だが =insert-file-contents-literally= が *prelude の
  delegating closure* で、prelude の =insert-file-contents= は (a) BEG/END 無視で
  whole-file (len 13)、(b) bridge の buffer model に *insert されない*
  (=buffer-string-len 0=)。=insert= 自体は bridge で動く。= 正しい
  =nelisp-ec-insert-file-contents= が bridge image で非 active = prelude fallback。
- *fix 候補*: (a) =nelisp-emacs-compat-fileio= の install を bridge image bake で
  active 化、(b) bridge 向け =insert= ベース range polyfill を bake。install 経路
  (backend dispatch) の特定が次の一手。
- *副産物*: pread64(17) buffer-free byte-exact range reader 実証済 (offset=4 len=4
  → (65 66 67 230))。大 dict の whole-file read 回避の最適化として後で有用 (今回は
  blocker でないため未 commit)。
- *着地 (2026-06-20)*: 修正は =nemacs-gui-file-bridge-runtime.el= 末尾に gated
  (=unless (fboundp 'nelisp-ec-insert-file-contents)= = bridge image 限定) で
  range-honoring =insert-file-contents(-literally)= を install。bridge image で
  cdb read pattern が (65 66 67 230) を返すことを確認、gui-bridge test +1。
  *full --batch stack は nelisp-ec が active なので gate skip = 完全無回帰*
  (insert-file-runtime-adapter / save-and-transform / find-file / skk
  keystroke-e2e 全 pass)。
- *重要発見 — 日本語入力 headline は既に動作*: skk は =nemacs-runtime-cdb.el= の
  独自 =cdb--read= を vendor-core image で使うため、GUI bridge の壊れた
  =insert-file-contents= に依存しない。skk E2E (=NEMACS_RUN_SKK_CONVERSION=1=) は
  *5/5 pass* (みらい→未来、はしる+SPC→走る 等)。= Doc 14 の「file-read が
  日本語入力の統一 blocker」は nemacs-runtime-cdb/skk 実装で既に解消済。本修正は
  GUI bridge の汎用 file-read 経路 (google-ime upstream 等) の correctness 改善。

** 3.3 B1 診断 (2026-06-20, 着手調査)

Doc 14 (2026-06-13) は「runtime 生 defun は登録されない (wrap-init 必須)」を
B1 の最重要 frontier としたが、その後実装が進み *stale*:

- *runtime macroexpand は解決済*: =--load FILE= / bridge =exec-runtime-image= /
  =eval= の全経路で =defun= / =defmacro= / =cl-defstruct= / =when-let= が load 時に
  展開・登録される。実 =dash.el= (148KB、複雑 macro 多用) を =--load= → =-map= /
  =-filter= が callable (=(-map (lambda (x)(* x x)) '(1 2 3))= = (1 4 9))。bridge
  でも dash =-map= 動作。
- *残 = runtime-loaded package が呼ぶ builtin の long-tail breadth* (B4 寄り):
  - bridge image は emacs-stub.el を bake しないため =save-match-data= /
    =match-data= / =set-match-data= が欠落 → s.el の =s-replace= 等が
    =void-function save-match-data= で abort。*本 wave で no-op stub を bridge に
    補完* (emacs-stub と同 behavior、=match-beginning/end= は bridge で real)。
    結果 =s-replace= / =s-join= / =s-upcase= が callable に。
  - =replace-match= の string 形は *bridge / full --batch 両方で nil* (= bridge
    固有でなく nelisp-emacs 全体の一般的限界)。s.el =s-trim= はこれ依存で未動作 =
    B1 スコープ外の別 issue。
- *macroexpand-all は未 fboundp* (単一 macroexpand は有り)。dash load には不要。

** 3.4 B3 / B4 診断 (2026-06-20, 着手調査)

Doc 11 §3.8 / §3.6 も B1/B2 同様に stale 気味:

- *B3 (process) は大半 SHIPPED 済*: full --batch で =make-process= /
  =start-process= / =call-process= / =shell-command= / =comint-mode= /
  =process-send-string= / =accept-process-output= が全て fboundp。Doc 11 §3.8
  の「shell-command + 基本 make-process の先」は既に越えている。残は
  =start-process-shell-command= (convenience wrapper) / PTY breadth 程度。
  本 wave では未着手 (実態確認のみ)。
- *B4 (callable promotion) = builtin breadth*: runtime macro 展開は B1 で解決済、
  package は load される。残るは個別 builtin の欠落で package 関数が callable
  にならないケース (B1 の save-match-data がその一例)。
  - *本 wave 増分 = replace-match (string 形)*: =emacs-search-builtins.el= に
    良い substrate-backed 実装 (=match-beginning/end= + concat) が既存だが、
    =emacs-stub-bulk.el:252= の mass-stub list が =replace-match= を no-op
    (=(&rest _) nil=) で先に bind し、=--install-function-p= gate
    (=fboundp= 済なら skip) で負けていた。gate に reader override
    (=(fboundp 'rdf)=) を追加し、全 search/match polyfill が reader で stub に
    勝つよう修正。bundle 再生成で full --batch でも =(s-trim "  hi  ")= = "hi"、
    s-replace / s-collapse-whitespace 動作。buffer 形 (STRING 無し) は旧 stub と
    同じ nil (no-op) を保ち無回帰。host は =rdf= 不在で C builtin 保護。
  - *残 breadth*: =macroexpand-all= 未実装、他 package が呼ぶ未実装 builtin の
    long-tail。実 package 駆動で逐次補完するのが筋。

* 5. 順序の根拠

- *小物先行*: (1) OS 判別で確立したテスト/bundle ワークフローを連続適用して
  習熟、(2) 各々 low-risk で degrade 安全、(3) A1/A2 は検出値の即応用で
  「色々と応用が可能」(user) を可視化。
- *中物の B2 を B 波先頭*: Doc 14 が file-read/buffer-ops を「日本語入力の
  統一 blocker」かつ「構造 blocker 解消済、残り I/O 網羅」と判定。効果最大。
- *大物は最後*: C2 self-host binary は他全項目の上位、C1 redisplay v2 は
  独立大レーン。B 波で daily-driver gate を「honest」(host fallback 無し,
  Doc 12:24-29) にしてから着手するのが筋。
