* 0. Purpose

This document changes the project framing from "a runnable Emacs
replacement first" to *reusable pure-Elisp Emacs-core libraries first*.

The original goal remains: run an Emacs-like editor on the NeLisp runtime
by replacing large parts of Emacs's C core with Elisp.  The next value
step is broader: make those C-core replacement libraries useful to other
projects before the full editor is complete.

Therefore, the project should temporarily optimize for:

1. stable library boundaries,
2. host-Emacs / NeLisp / standalone portability,
3. adapter separation,
4. package-level tests and documentation,
5. reintegrating =nelisp-gui= under this tree while the shared library
   contracts mature.

Only after the library surface is stable should the GUI be split out
again as an external consumer.

* 1. Decision

Adopt a *library-first monorepo phase*.

During this phase, =nelisp-emacs= owns the reusable Emacs-core Elisp
libraries, the =nemacs= application assembly, and the GUI adapter code
that consumes those libraries.  =nelisp-gui= should be brought back under
this repository as a workspace component, not because GUI belongs forever
inside this repo, but because cross-repository boundaries are still too
expensive while the library APIs are moving.

The desired final shape is:

#+begin_src text
nelisp-emacs/
  packages/
    foundation/        reusable pure Elisp primitives
    buffer/            buffer, point, text properties, editing substrate
    editor-core/       window, frame, keymap, command-loop, minibuffer
    io/                file, process, shell, network, sqlite adapters
    display/           redisplay model, TUI backend, event parser
    gui-adapter/       GUI transport adapters consuming editor-core
  apps/
    nemacs/            runnable editor assembly
  src/                 current source location during migration
  test/                current tests, gradually grouped by package
#+end_src

Physical movement into =packages/= is optional at first.  Logical package
boundaries and dependency rules are mandatory before large file moves.

* 2. Goals

** 2.1 Reuse goals

- Other Elisp projects can depend on a subset such as foundation,
  buffer, or command-loop without pulling in TUI / GUI / =nemacs-main=.
- Host Emacs can load the reusable libraries without replacing host C
  primitives unless an explicit compatibility shim is requested.
- NeLisp can load the same libraries in standalone mode, with runtime
  adapters filling the missing primitive layer.
- Libraries expose prefixed APIs first.  Unprefixed Emacs-compatible
  names are installed by explicit shim modules.
- Tests prove package behavior independently from the full editor app.

** 2.2 Project goals

- Make =nelisp-emacs= valuable even before it is a full Emacs
  replacement.
- Turn current C-core replacement work into reusable artifacts.
- Reduce duplicate behavior in GUI/TUI bridges by forcing adapters to
  consume the same shared runtime modules.
- Make future repository split mechanical: the GUI split should become a
  consumer extraction, not a behavioral rewrite.

* 3. Non-goals

- Do not immediately move every file into a package directory.
- Do not publish unstable APIs as if they were final.
- Do not let =nemacs-main.el= or GUI bridge code become the owner of
  command semantics.
- Do not make the host-Emacs compatibility shim the only API.  The
  prefixed library API must remain usable without global namespace
  takeover.
- Do not add new NeLisp runtime primitives merely to make extraction
  easier.  Missing substrate should be documented and adapter-gated.

* 4. Library taxonomy

** 4.1 Foundation

Pure Elisp primitives and compatibility helpers with minimal global editor
state.

Current candidates:

- =emacs-fns.el=
- =emacs-list.el=
- =emacs-string.el=
- =emacs-symbol.el=
- =emacs-hash.el=
- =emacs-numeric.el=
- =emacs-error.el=
- =emacs-backquote.el=
- =emacs-pcase.el=
- =emacs-cl-macros.el=
- =emacs-time.el=
- =emacs-subr-extras.el=
- =seq.el=, =map.el=, =subr-x.el=, =cl-lib.el= shims

Extraction rule: these modules should not depend on buffer/window/frame,
TUI, GUI, =nemacs-main=, or process state.

** 4.2 Buffer and editing substrate

Reusable editor data model: buffers, point, text properties, basic edit
operations, line/search/edit builtins, and undo records.

Current candidates:

- =nelisp-emacs-compat.el=
- =nelisp-text-buffer.el=
- =emacs-buffer.el=
- =emacs-buffer-builtins.el=
- =emacs-line-builtins.el=
- =emacs-search-builtins.el=
- =emacs-edit-builtins.el=
- =emacs-undo.el= / =emacs-undo-builtins.el=

Extraction rule: buffer semantics belong here, not in GUI/TUI adapters.

** 4.3 Editor core

Emacs command semantics above the buffer model.

Current candidates:

- =emacs-window.el= / =emacs-window-builtins.el=
- =emacs-frame.el= / =emacs-frame-builtins.el=
- =emacs-keymap.el= / =emacs-keymap-builtins.el=
- =emacs-command-loop.el= / =emacs-command-loop-builtins.el=
- =emacs-minibuffer.el= / =emacs-minibuffer-builtins.el=
- =emacs-mode.el= / =emacs-mode-builtins.el=
- =emacs-faces.el= / =emacs-faces-builtins.el=
- =emacs-help.el=, =emacs-info.el=

Extraction rule: this layer may depend on foundation and buffer, but not
on a concrete GUI/TUI transport.

** 4.4 IO and runtime adapters

Boundary modules that abstract host Emacs, NeLisp, standalone, file,
process, network, sqlite, and shell behavior.

Current candidates:

- =emacs-fileio.el= / =emacs-fileio-builtins.el=
- =emacs-callproc.el=
- =emacs-process.el= / =emacs-process-builtins.el=
- =emacs-shell-command.el=
- =emacs-network-ffi.el=
- =emacs-sqlite.el= / =emacs-sqlite-ffi.el=
- =emacs-standalone.el=
- =nemacs-runtime-*.el=

Extraction rule: adapters may be runtime-specific, but reusable libraries
must not silently require a specific adapter.

** 4.5 Display libraries

Display model and terminal backend.

Current candidates:

- =emacs-redisplay-core.el=
- =emacs-redisplay.el=
- =emacs-redisplay-builtins.el=
- =emacs-tui-backend.el=
- =emacs-tui-event.el=
- =emacs-tui-terminfo.el=

Extraction rule: redisplay should expose a reusable render model.  TUI is
one backend.  GUI should consume the same render/editor state instead of
forking command behavior.

** 4.6 Application assembly

The runnable editor application.

Current candidates:

- =nemacs-loadup.el=
- =emacs-init.el=
- =nemacs-main.el=
- =bin/nemacs=
- bootstrap / runtime image scripts

Extraction rule: application files compose libraries and adapters.  They
should not become the primary implementation location for reusable Emacs
semantics.

* 5. nelisp-gui reintegration

** 5.1 Rationale

Keeping =nelisp-gui= outside while core APIs are still shifting creates
friction:

- behavior fixes must be coordinated across repositories,
- GUI may grow bridge-local copies of command semantics,
- library contracts are harder to validate against a real consumer,
- refactors are delayed because the downstream breakage is hidden.

Temporarily reintegrating GUI code makes the GUI a first-class consumer
test for the reusable libraries.  The goal is not to make GUI code a
permanent dependency of the core; the goal is to harden the core by
keeping a demanding consumer nearby.

** 5.2 Placement

Preferred temporary placement:

#+begin_src text
gui/
  README.org
  src/
  test/
  scripts/
#+end_src

The =gui/= tree must depend inward on library packages.  Library packages
must not depend outward on =gui/=.

** 5.3 Reintegration rules

- GUI may own display protocol, window-system resources, raw input
  decoding, transport setup, and painting.
- GUI must not own buffer mutation semantics, command dispatch, keymap
  lookup, minibuffer completion, undo, kill-ring, file command semantics,
  or window layout decisions.
- If GUI needs a behavior that is currently missing in shared runtime,
  implement it in the shared library first, then call it from GUI.
- GUI-local fallback code must be tracked as deletion candidates.

** 5.4 Re-split criteria

Split GUI out again only when all are true:

1. GUI consumes published package APIs, not private file internals.
2. The shared library package tests pass without GUI.
3. GUI tests pass as a downstream consumer.
4. Bridge-local semantic fallbacks are zero or explicitly documented.
5. Package dependency graph is documented and enforced in CI.
6. =AGENTS.md= and =CLAUDE.md= rules are satisfied for at least one full
   release cycle.

* 6. API policy

** 6.1 Public API classes

| Class | Naming | Stability | Consumer |
|-------+--------+-----------+----------|
| Library public | prefixed, e.g. =emacs-buffer-*= / =nelisp-ec-*= | target stable | external projects |
| Compatibility shim | unprefixed, e.g. =current-buffer= | explicit opt-in | Emacs-compatible packages |
| Shim helper | prefixed helper in =emacs-*-builtins.el= | compatibility implementation detail unless facade-promoted | shim modules |
| Validation helper | prefixed guard helper, e.g. =*-check-live= | internal unless facade-promoted | package internals |
| Adapter API | =*-backend= / =*-gui= / =*-tui= | backend-specific | app/adapters |
| Internal | =--= names | unstable | same module only |

** 6.2 Shim rule

Host-Emacs global names must be installed only by explicit shim modules or
by carefully gated compatibility files.  A reusable library should prefer
prefixed functions and data structures so another project can load it
without changing global Emacs behavior.

Prefixed helpers defined inside =emacs-*-builtins.el= are shim helpers by
default, even though their names look public.  They may appear in the
generated package API inventory, but they should not be treated as stable
consumer APIs unless they are explicitly added to the facade stable API
manifest.  Prefer the owning substrate API, such as =nelisp-ec-*= or
=emacs-buffer-*=, for new reusable integrations.

Prefixed validation helpers such as =*-check-live= are internal guard
surface by default.  Promote them only when an external consumer needs the
guard behavior itself, not merely because package internals call it.

** 6.3 Internal dependency rule

External consumers must never rely on =--= functions.  During the
library-first phase, any cross-module use of =--= functions should be
treated as a design smell and either promoted to a public prefixed helper
or localized.

* 7. Dependency rules

Allowed direction:

#+begin_src text
foundation
  -> buffer/edit
  -> editor-core
  -> display model
  -> adapters
  -> apps/gui/tui
#+end_src

Forbidden direction:

- foundation -> buffer/window/frame/process/display
- buffer -> GUI/TUI application code
- editor-core -> concrete GUI transport
- display model -> =nemacs-main= shortcuts
- reusable package -> =apps/nemacs= or =gui/=

* 8. Test strategy

Each package group needs a small, fast gate:

| Group | Minimal gate |
|-------+--------------|
| foundation | pure host ERT + NeLisp reader smoke when substrate-sensitive |
| buffer/edit | host ERT around buffer workflows + standalone smoke |
| editor-core | focused keymap/minibuffer/command-loop/window tests |
| IO adapters | host/standalone split tests; no silent no-op success |
| display | redisplay core unit tests + backend smoke |
| GUI | consumer tests proving no duplicate command semantics |
| app | =make verify-nemacs-daily-driver= and completion gate |

The existing =make test-fast= remains the default development gate.  The
goal is to grow package-selective gates underneath it, not replace it.

* 9. Migration plan

** Phase L0 -- Document and enforce rules

- Add this design doc.
- Add root =AGENTS.md= and =CLAUDE.md= so Codex / Claude both follow the
  library-first policy.
- Treat new app-local semantics as review findings.

** Phase L1 -- Logical package map

- Add a package map document or table generated from =emacs-init.el=.
- Mark each =src/*.el= file with one group: foundation, buffer,
  editor-core, IO, display, adapter, app, vendor shim, or legacy.
- Do not move files yet.

** Phase L2 -- API audit

- For each package group, list public prefixed functions and accidental
  cross-module private calls.
- Promote only the helpers needed by real consumers.
- Keep unprefixed Emacs compatibility installs in shim modules.

** Phase L3 -- Adapter cleanup

- Move command semantics out of =nemacs-main.el=, GUI bridge, and TUI
  shortcuts into shared runtime modules.
- Keep adapter files thin: translate input/output, call shared behavior,
  render returned state.

** Phase L4 -- Reintegrate GUI

- Import =nelisp-gui= under =gui/=.
- Wire it as a consumer of shared runtime modules.
- Add inventory gates for GUI-local semantic fallbacks.

** Phase L5 -- Package layout

- Once logical boundaries hold, optionally move files into =packages/=.
- Add package descriptors / generated load files.
- Keep compatibility aliases for old require names during transition.

** Phase L6 -- Consumer validation

- Build small example consumers:
  - buffer library only,
  - command-loop/minibuffer only,
  - redisplay/TUI only,
  - GUI consumer.
- Validate that each does not pull in =nemacs-main=.

** Phase L7 -- Re-split GUI and publish

- Split GUI out only after §5.4 criteria are met.
- Publish reusable package artifacts first; publish applications as
  consumers.

* 10. Review checklist

Every non-trivial change should answer:

1. Which package group owns this behavior?
2. Is the API prefixed and reusable?
3. Does it require host Emacs, NeLisp, or a specific adapter?
4. Did this add command semantics to an app or GUI file?
5. Is there a focused package-level test?
6. Does =make test-fast= still represent the right first gate?
7. If GUI/TUI was touched, is fallback behavior tracked for deletion?

* 11. Success criteria

The library-first phase is succeeding when:

- a new project can load foundation + buffer without loading =nemacs-main=,
- command-loop/minibuffer can be exercised independently,
- GUI and TUI call the same shared command semantics,
- unprefixed compatibility shims are opt-in,
- public APIs are documented by package group,
- package tests catch regressions before full app gates,
- extracting GUI again would mostly change repository layout, not behavior.
