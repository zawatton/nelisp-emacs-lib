* Status — read this first

This is an *early, exploratory research project*.  It is *not* a
usable Emacs replacement and is unlikely to become one for several
years.  Most user-visible Emacs features are absent, the API is
unstable, and breaking changes are expected on most commits.

Concretely, as of 2026-05-15:

- Phase 1 (= core editor substrate: =buffer.c=, =window.c=, =frame.c=,
  =keymap.c=, =minibuffer.c= ported to elisp) has its *§4 sub-tasks
  + close mini-demo shipped*.  The five core modules expose a Phase 1
  API surface that runs end-to-end in batch ERT — see
  =demo/phase1-close-demo.el= for the integration shape.  Advanced
  text-property (Phase 9c), full redisplay, and frame backend
  swap-in remain future phases.
- Phase 2 (= TUI reference display backend) has *scaffolding only*.
  The Phase 1 close demo uses =emacs-tui-stub.el= as a no-op canvas
  backend; Phase 2 swaps in =emacs-tui-backend.el= +
  =emacs-tui-terminfo.el= + =emacs-tui-event.el=.
- Phases 3-5 (= ~xdisp.c~ port, MELPA shim, Emacs bootstrap /
  self-host) are *unstarted*.

The repository ships *597 ERT* across the Phase 1 modules, all
green; integration round-trip is exercised in seven additional
=phase1-close-demo-*= tests.

If you are looking for a working pure-elisp Emacs binary today, this
is not it.  See [[#long-term-roadmap][Long-term roadmap]] below for an
honest timescale.

* What is this?

=nelisp-emacs= is the Layer 2 (extension package) + Layer 3 (TUI
reference display backend) of the
[[https://github.com/zawatton/nelisp][NeLisp]] stack.  The long-term
goal is to re-implement Emacs's core C code in elisp, running on the
NeLisp runtime, so that a pure-elisp Emacs binary can be produced
without a hand-written C core.

* Relationship to NeLisp

- =Layer 1 (NeLisp core runtime)= = upstream
  [[https://github.com/zawatton/nelisp][~nelisp~]] (vendored under
  =vendor/nelisp/= via =bin/build-nelisp=).  This repo never modifies
  Layer 1.
- =Layer 2 (extension package)= = partly upstream (the
  =nelisp-emacs-compat=, =nelisp-process=, =nelisp-regex=,
  =nelisp-coding=, =nelisp-sqlite=, =nelisp-base64=,
  =nelisp-secure-hash=, =nelisp-json=, =nelisp-filenotify=, …
  modules are part of the NeLisp distribution).  *This repo is the
  remaining Layer 2 work* — the Emacs interactive editor substrate
  (buffer / window / frame / keymap / minibuffer / etc.) ported
  from Emacs C.
- =Layer 3 (display backend)= = TUI primary (in this repo).  The
  GTK4 GUI sibling lives in
  [[https://github.com/zawatton/nelisp-gtk][~nelisp-gtk~]] (currently a
  Rust boot-stub + elisp policy split, with a long-term goal of
  collapsing the Rust layer in favour of pure-elisp NeLisp-native
  GTK4 FFI bindings — see that repo's README "Long-term goal" section
  for the B0/B/C plan).  Cocoa / Win32 / =wgpu= backends are open to
  community contribution.

* Repository layout

- =src/= — Layer 2 / Layer 3 elisp implementation
  (=emacs-buffer.el=, =emacs-window.el=, =emacs-frame.el=,
  =emacs-keymap.el=, =emacs-minibuffer.el=,
  =emacs-tui-backend.el=, =emacs-tui-terminfo.el=,
  =emacs-network-ffi.el=, =emacs-sqlite-ffi.el=,
  =emacs-server-polyfills.el=, …)
- =test/= — ERT tests for the modules under =src/= plus the
  =phase1-close-demo-test.el= integration battery
- =demo/= — runnable scenarios (=phase1-close-demo.el= drives the
  Phase 1 close gate; invoke via =make demo=)
- =vendor/emacs-lisp/= — vendored copies of selected Emacs
  =lisp/= subtrees used as references during the port
  (e.g. =vendor/emacs-lisp/keymap.el= is the upstream source for
  =key-parse= / =key-valid-p= delegated by =emacs-keymap.el='s
  newer-API section; =vendor/emacs-lisp/org/= for the day when an
  =org-mode= bootstrap becomes meaningful)
- =vendor/nelisp/= — produced by =bin/build-nelisp= (= =make nelisp=)
- =docs/design/= — design documents for each port phase
- =bench/= — micro-benchmarks (advisory; not merge gates)

The =vendor/= elisp is *imported, not implemented*.  Until the Layer
2/3 substrate is functional enough to host those packages, the vendored
copies are reference material — they will not produce a working
=org-mode= or =Org agenda= on top of =nelisp-emacs= today.

* Initial scope (Phase 1)

Per =docs/design/01-architecture.org= (LOCKED).  Five core modules +
a TUI stub form the Phase 1 surface; the real TUI backend is Phase 2.
The §4 close-mini-demo (Doc 34 §9.5 / gate-M9) ties them together.

1. =src/emacs-buffer.el= = =buffer.c= port (buffer-local
   variables, text-property MVP + =category= symbol inheritance +
   change navigation, overlay primitives, undo MVP, modification
   tracking).  Doc 41 advanced text-property and the full undo
   contract remain future phases.
2. =src/emacs-window.el= = =window.c= port (query, split / delete,
   size / position, window-local config, selection,
   newer-API helpers: =split-window-below/right=, =other-window=,
   =window-live-p=, =enlarge-window= / =shrink-window=).
3. =src/emacs-frame.el= = =frame.c= port (Phase 1 stub mode per
   Doc 34 §2.11; the real backend swap-in is Phase 2 / Doc 43 §2.1).
4. =src/emacs-keymap.el= = =keymap.c= port (7段 / opt-in 9段 chain,
   parent inheritance, prefix bindings, newer kbd-style API
   delegating =key-parse= / =key-valid-p= to
   =vendor/emacs-lisp/keymap.el=).
5. =src/emacs-minibuffer.el= = =minibuffer.c= port (line readers,
   typed readers, =completing-read= with =completion-ignore-case=
   support, =try-/all-/test-completion= helpers, plug-in reader for
   ERT).
6. =src/emacs-tui-stub.el= = TUI stub backend (Phase 1 close gate
   only).  Provides the frame / canvas / capability surface so the
   §4 demo runs end-to-end without ncurses; replaced by the real
   =src/emacs-tui-backend.el= in Phase 2.

* Long-term roadmap
:PROPERTIES:
:CUSTOM_ID: long-term-roadmap
:END:

These estimates are *aspirational*, not commitments.  Each phase
boundary is a real piece of work — the figures below assume single
maintainer evening hours.

| Phase   | Scope                              | LOC estimate | Wall-clock | Status         |
|---------+------------------------------------+--------------+------------+----------------|
| Phase 1 | Layer 2 core (buffer / window / …) | ~5-10K       | 6-12 mo    | §4 + close demo shipped (2026-05-15) |
| Phase 2 | TUI backend reference impl         | ~5-10K       | 6-12 mo    | scaffolding    |
| Phase 3 | =xdisp.c= port (display engine)    | ~10-30K      | 12-24 mo   | unstarted      |
| Phase 4 | MELPA compat layer (shim)          | ~20-50K      | 1-2 yr     | unstarted      |
| Phase 5 | Emacs bootstrap (self-host)        | full         | 3-5 yr     | unstarted      |

In particular: =org-mode= depends on Phase 1 (buffer + text
properties) + Phase 2 (TUI redisplay) + Phase 3 (display engine, for
visual folding / agenda / table alignment) all reaching usable
state.  Today, ~ert-run~ batch-style use of vendored elisp is
plausible; interactive editing in =nelisp-emacs= is not.

* Building

#+begin_src sh
make nelisp        # populate vendor/nelisp/ from upstream
make compile       # byte-compile src/
make test          # run the ERT test suite
make demo          # run the Phase 1 close mini demo and print result
#+end_src

The nelisp binary tool is the source of truth for
=vendor/nelisp/=; do not add Makefile targets or scripts that assume
an external nelisp checkout outside =$NEMACS_HOME=.

=make demo= prints the demo's observable state plist so you can
inspect the integration round-trip from the shell — useful as a
smoke test on a fresh clone.
