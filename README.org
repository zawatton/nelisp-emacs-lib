* Status — read this first

This is an *early, exploratory research project*.  It is *not* a
usable Emacs replacement and is unlikely to become one for several
years.  Most user-visible Emacs features are absent, the API is
unstable, and breaking changes are expected on most commits.

Concretely, as of 2026-05-27:

- Phase 1 (= core editor substrate: =buffer.c=, =window.c=, =frame.c=,
  =keymap.c=, =minibuffer.c= ported to elisp) has its *§4 sub-tasks
  + close mini-demo shipped*.  The five core modules expose a Phase 1
  API surface that runs end-to-end in batch ERT — see
  =demo/phase1-close-demo.el= for the integration shape.  Advanced
  text-property (Phase 9c), full redisplay, and frame backend

* What is this?

=nelisp-emacs= is the Layer 2 (extension package) + Layer 3 (TUI
reference display backend) of the
[[https://github.com/zawatton/nelisp][NeLisp]] stack.  The long-term
goal is to re-implement Emacs's core C code in elisp, running on the
NeLisp runtime, so that a pure-elisp Emacs binary can be produced
without a hand-written C core.

* Relationship to NeLisp

- =Layer 1 (NeLisp core runtime)= = upstream
  [[https://github.com/zawatton/nelisp][~nelisp~]] (vendored under
  =vendor/nelisp/= via =bin/build-nelisp=, or supplied by a sibling
  checkout via =NELISP_HOME=).  NeLisp provides the pure-Elisp
  standalone reader at =target/nelisp-standalone-reader=, including
  =repl [--no-prompt]= for persistent evaluator development.
- =Layer 2 (extension package)= = partly upstream (the
  =nelisp-emacs-compat=, =nelisp-process=, =nelisp-regex=,
  =nelisp-coding=, =nelisp-sqlite=, =nelisp-base64=,
  =nelisp-secure-hash=, =nelisp-json=, =nelisp-filenotify=, …
  modules are part of the NeLisp distribution).  *This repo is the
  remaining Layer 2 work* — the Emacs interactive editor substrate
  (buffer / window / frame / keymap / minibuffer / etc.) ported
  from Emacs C.
- =Layer 3 (display backend)= = TUI primary (in this repo).  The
  GTK4 GUI sibling lives in
  [[https://github.com/zawatton/nelisp-gtk][~nelisp-gtk~]].  Cocoa / Win32 /
  =wgpu= backends are open to community contribution.

The =vendor/= elisp is *imported, not implemented*.  Until the Layer
2/3 substrate is functional enough to host those packages, the vendored
copies are reference material — they will not produce a working
=org-mode= or =Org agenda= on top of =nelisp-emacs= today.

* Initial scope (Phase 1)

Per =docs/design/01-architecture.org= (LOCKED).  Five core modules +
a TUI stub form the Phase 1 surface; the real TUI backend is Phase 2.
The §4 close-mini-demo (Doc 34 §9.5 / gate-M9) ties them together.

1. =src/emacs-buffer.el= = =buffer.c= port (buffer-local
   variables, text-property MVP + =category= symbol inheritance +
   change navigation, overlay primitives, undo MVP, modification
   tracking).  Doc 41 advanced text-property and the full undo
   contract remain future phases.
2. =src/emacs-window.el= = =window.c= port (query, split / delete,
   size / position, window-local config, selection,
   newer-API helpers: =split-window-below/right=, =other-window=,
   =window-live-p=, =enlarge-window= / =shrink-window=).
3. =src/emacs-frame.el= = =frame.c= port (Phase 1 stub mode per
   Doc 34 §2.11; Phase 2 added the TUI backend swap-in per Doc 43
   §2.1 plus the SIGWINCH and stdin keyboard wire-ups so the
   substrate can drive a real terminal once redisplay lands).
4. =src/emacs-keymap.el= = =keymap.c= port (7段 / opt-in 9段 chain,
   parent inheritance, prefix bindings, newer kbd-style API
   delegating =key-parse= / =key-valid-p= to
   =vendor/emacs-lisp/keymap.el=).
5. =src/emacs-minibuffer.el= = =minibuffer.c= port (line readers,
   typed readers, =completing-read= with =completion-ignore-case=
   support, =try-/all-/test-completion= helpers, plug-in reader for
   ERT).
6. =src/emacs-tui-stub.el= = TUI stub backend (Phase 1 close gate
   only).  Provides the frame / canvas / capability surface so the
   §4 demo runs end-to-end without ncurses; replaced by the real
   =src/emacs-tui-backend.el= in Phase 2.

* Phase 2 substrate (TUI reference impl, Doc 43 §3.1 Phase 11.A)

Phase 2 adds the real TUI substrate sitting under =emacs-frame.el='s
backend dispatch table.  The close gate (commits 30d261f → 710c01c)
ships these modules:

1. =src/emacs-tui-backend.el= = curses-style canvas backend.  Owns
   the per-frame ANSI escape stream, cursor, scroll region, color
   model, and capability registry.  Driven through the
   =emacs-frame--backend-dispatch= plist that
   =emacs-frame-use-tui-backend= installs.
2. =src/emacs-tui-terminfo.el= = capability detection from =TERM= /
   =COLORTERM= env values.  Table-driven (=xterm-256color=,
   =xterm-kitty=, =alacritty=, =screen-256color=, …) plus a
   =extra-color-terminals= heuristic that escalates well-known
   truecolor emulators when =COLORTERM= is unset.
3. =src/emacs-tui-event.el= = stdin parser + event queue.  Decodes
   CSI / SS3 escape sequences into key plists
   (=(:type key :name NAME :modifiers MODS)=), tracks SIGWINCH via
   =window-size-change-functions=, and exposes a pull-on-demand
   =event-poll= API (Doc 43 §2.6).
4. =emacs-frame.el= integration (T159 / T160 / T161) =
   =emacs-frame-use-tui-backend= swaps the dispatch table,
   installs a SIGWINCH callback that resizes every live frame +
   runs =emacs-frame-tui-resize-hook=, and bridges the TUI event
   source into =emacs-keymap--read-event-fn= so
   =read-key-sequence= consumes real terminal input.
   =emacs-frame-use-stub-backend= reverts everything (including
   restoring a caller's prior =read-event-fn=).
5. =demo/phase2-close-demo.el= + =test/phase2-close-demo-test.el=
   = the end-to-end ship gate.  Drives the full pipeline (TUI swap
   → terminfo → mini find-file → keystroke feed-bytes → keymap
   reader → buffer mutation → SIGWINCH → mini save-buffer → stub
   revert) and asserts the observable state, including 3-env
   validation under =xterm-256color= / =xterm-kitty= /
   Windows-Terminal-style =COLORTERM=truecolor=.

What Phase 2 does *not* yet ship:
- Live execution on a real device.  The substrate is verified
  through env-injection (=:env= plist forwarded to
  =emacs-tui-terminfo-detect=); real-tty validation on physical
  =xterm-256color= / kitty / Windows Terminal sessions is part of
  the Phase 5 bootstrap work.
- Modifier folding on symbol keys (=C-<up>=, =M-<f1>=, …).  Phase 2
  forwards the modifier list through =emacs-frame-tui-key-hook= but
  the keymap reader returns the bare symbol; symbol-name modifier
  encoding lands when a real command loop needs it.

* Phase 3 substrate (redisplay engine MVP, Doc 43 §3.2 Phase 11.B)

Phase 3 adds the redisplay engine that drives buffer contents onto
the TUI canvas.  Per Doc 43 §3.2 the MVP scope is the
redisplay-driver + glyph-matrix (face-realize and
display-property-handler are v2.x incremental).  The close gate
shipped 2026-05-06; the following commits build it up:

1. =src/emacs-redisplay.el= (~1830 LOC) = single-file MVP carrying
   the redisplay handle, glyph-matrix struct, driver loop,
   row-hash diff, fingerprint short-circuit, per-row incremental
   rebuild, buffer-string text-tick cache, and the TUI canvas
   flush.  Phase 3.B.1 face-realize MVP, 3.B.2 overlay
   before/after-string, 3.B.3 256-color/truecolor SGR, 3.B.4
   mode-line + invisible + row-hash diff, 3.B.5 fingerprint
   throughput gate, 3.B.6 row-incremental rebuild, 3.B.7
   buffer-string cache are all merged.
2. =test/emacs-redisplay-test.el= = 80 ERT covering driver / matrix
   / face / overlay / scroll / mode-line / scroll re-fill /
   text-property face change propagation / p99 latency.
3. =bench/bench-redisplay.el= + =make bench= target = persistent
   throughput + p99 latency verification.  3 scenarios (static /
   edit / scroll) × 2 modes (:diff vs :full), per-iter sample +
   p50/p99/p99.9/max + gate verdict.
4. =emacs-buffer.el= integration: =put-text-property= /
   =add-text-properties= / =remove-text-properties= bump
   =modified-tick=, =nelisp-ec-insert= / =-delete-region= /

make demo            # run the Phase 1 close mini demo and print result
make demo-phase2     # run the Phase 2 close mini demo and print result
#+end_src

For regular development, =make test= stays on the host Emacs driver
and keeps the suite fast.  The =--driver=nelisp= path now targets the
pure-Elisp NeLisp stack and is much slower, so it is exposed as explicit
smoke targets.  =make doctor= runs a fast readiness check for the clean
host batch path plus the resolved NeLisp driver.  It checks literal and
list-form eval through the standalone reader's exit-code contract.  If
this fails before =nemacs-main= is entered, the next fix belongs in the
driver/runtime layer rather than the editor bootstrap.
=make build-nelisp-bootstrap=
creates a plain Elisp preload bundle at =build/nemacs-bootstrap.el=;
=bin/nemacs --driver=nelisp= uses it automatically when present.  This is
a source preload bundle, not a pdump/AST image.  Set
=NEMACS_BOOTSTRAP_BUNDLE=none= to force source-by-source cold loading.

=make verify-nelisp-standalone= is the current pure standalone-reader
gate.  It runs =make doctor=, =make test-nelisp=,
=make test-nelisp-runtime-image=, =make verify-vendor-class-a=, and
=make verify-vendor-core=, proving the reader binary, the
=bin/nemacs --driver=nelisp --batch= boot script path, the source-v1
runtime-image bake/replay path, the lightweight Class-A vendor smoke, and
the current daily-driver vendor-core smoke without requiring the legacy
command path.

=make bake-image= still builds the older =build/nemacs-loadup.nli=
state image through =emacs-dump= / =image-loader= for compatibility
with =--load-image=.  The legacy loader is lazy and is not required by
=nemacs-main= unless =:images= / =--load-image= is actually used; it is
not the daily-driver startup path because it does not freeze the
NeLisp runtime heap or function environment.

=make bake-runtime-image= writes a NeLisp =.nlri= runtime image at
=build/nemacs-runtime.nlri= using the resolved =NELISP_BIN=
=dump-runtime-image= entry.  Current NeLisp standalone-reader builds
provide the source-v1 =dump-runtime-image= / =extend-runtime-image=
command surface.  With the source-v1 image format, that file is a
replayable source bundle: loading it evaluates the stored bootstrap forms
again in a fresh NeLisp environment.  It is therefore useful as a shared
runtime-image entry point and for future snapshot-compatible preload
lanes, but it should not be treated as a startup accelerator yet.

On the standalone-reader path, =bin/nemacs --runtime-image FILE= now
uses the direct standalone =exec-runtime-image= command.  The
standalone reader builds direct exec/eval image commands into the same
multi-form source replay path used by normal file evaluation.  Direct
=exec-runtime-image= appends every supplied =FORM...= followed by an
explicit zero exit sentinel; direct =eval-runtime-image= evaluates the
supplied =FORM...= as a =progn=, prints the final value, and then exits
zero.  The batch runtime-image smoke is part of =make
verify-nelisp-standalone=; the interactive and vendor-core image smokes
keep explicit longer bake/replay timeouts and remain opt-in diagnostics.

The runtime-image bake recipes share
=scripts/nemacs-runtime-image-preload.el=, which keeps preload policy
out of the Makefile's quoted one-liners.  =make
bake-interactive-runtime-image= preloads the pure-Elisp TUI backend,
TUI event parser, lightweight first-frame redisplay core, and nemacs
command keymap.  The full =emacs-redisplay.el= engine remains lazy;
first TUI realisation uses =emacs-redisplay-core.el= so startup does
not pay the full redisplay source load before a frame exists.  Use
=NEMACS_RUNTIME_IMAGE=auto= to let =bin/nemacs --driver=nelisp= choose
the batch or interactive image by mode when it exists.

=make bake-vendor-core-runtime-image= uses the resolved NeLisp
=extend-runtime-image= command: it reads =build/nemacs-runtime.nlri= as
the base image and writes =build/nemacs-vendor-core-runtime.nlri= without
overwriting the base image.  If =build/nemacs-runtime.nlri= already
exists, this target extends that image directly instead of forcing a
fresh bootstrap dump; it only invokes the base-image bake when the base
file is missing.  On standalone-reader source-v1 images, extension writes
the concatenated source bundle and leaves full replay validation to the
wrapper smoke, because nested command-frame validation is not the stable
path for images that contain `load'.  The first extension lane directly
installs the daily-driver =files= and =simple= shim surface from the
already-loaded preload helper, avoiding slow source =require= for those
two shims.
=test-nelisp-vendor-core-runtime-image= is still an experimental
long-timeout smoke rather than a daily gate.  The direct vendor-core
surface currently covers =files=, =simple=, and the Doc 03 Phase 1
daily-driver command facades for =dired=, =help-mode=, =help-fns=,
=lisp-mode=, =ielm=, =isearch=,
=minibuffer=, and =project=.  It also directly installs the lightweight
support and Class-A utility/i18n surfaces for =subr-x=, =seq=, =map=,
=lisp=, =case-table=, =cdl=, =range=, =regi=, =hex-util=, =map-ynp=,
=charprop=, =charscript=, =emoji-labels=, =iso-transl=, =cp51932=,
=eucjp-ms=, =fontset=, =idna-mapping=, and =ja-dic-utl=.  In host
smoke, running the vendor-core extension and then =VENDOR_CORE_LIMIT=0=
now proves all 29 candidates without source =require= during extension.

=make verify-vendor= includes the generated vendor inventory, a small
Class-A load lane, and a strict daily-driver core lane.  Both smoke lanes
are part of the pure standalone-reader gate.  The vendor-core lane uses a
  temporary file-based boot script for the standalone reader.  By default the
Class-A lane proves all eighteen lightweight modules
(=case-table=, =cdl=, =backquote=, =lisp-float-type=,
=hex-util=, =lisp=, =map-ynp=, =range=, =regi=,
=charprop=, =charscript=, =emoji-labels=, =iso-transl=,
=cp51932=, =eucjp-ms=, =fontset=, =idna-mapping=, and
=ja-dic-utl=), while the
core lane now defaults to =VENDOR_CORE_LIMIT=0=, proving all current
daily-driver vendor-core candidates under the standalone reader.
The same runner already has Doc 03 Phase 1 candidates for =simple=, =dired=,
=help-mode=, =help-fns=, =subr-x=, =seq=, =map=, =lisp=, =case-table=, =cdl=, =range=, =regi=, =lisp-mode=,
=ielm=, =isearch=, =minibuffer=, =project=, plus the Class-A utility/i18n
facades =hex-util=, =map-ynp=, =charprop=, =charscript=, =emoji-labels=,
=iso-transl=, =cp51932=, =eucjp-ms=, =fontset=, =idna-mapping=, and
=ja-dic-utl=.  Set =VENDOR_CORE_LIMIT=N= to run only the first N
modules when narrowing a regression.  Use
=VENDOR_CORE_MODULES=simple= or a comma/space-separated list to diagnose
a candidate without first paying the already-verified =files= lane.  The
=simple= lane is currently satisfied by

=emacs-isearch=, =emacs-minibuffer-builtins=, and =emacs-project=.
The =subr-x= lane is satisfied by =src/subr-x.el=, which collects the

=lisp= is satisfied by =src/lisp.el=, which gives NeLisp a standard

=emacs-lisp/lisp.el= does not provide one.  The facade covers the

=cdl= is satisfied by =src/cdl.el=, a small standard-name facade over

=charprop= is satisfied by =src/charprop.el=, a lightweight

=profile-nelisp-bootstrap= now uses host Emacs to split the generated
=build/nemacs-bootstrap.el= bundle into source sections, then runs the
standalone reader on cumulative prefixes and reports host-measured wall
time for each section.  Set =NELISP_BOOT_PROFILE_LIMIT=N= to stop after
the first N bootstrap sections.  =diagnose-vendor-form-walk= likewise
uses host Emacs to parse a vendor file into top-level forms, then runs
the standalone reader on cumulative prefixes with a success sentinel.
=diagnose-vendor-load-replay= goes one step closer to real loading: it
generates a temporary standalone-reader script that loads the standalone
stdlib prelude (=VENDOR_LOAD_PRELUDE=), the generated bootstrap bundle,
and then the files in =VENDOR_LOAD_FILES= as top-level =load= forms.  It
then requires =VENDOR_LOAD_PROOF_FORM= to become true before returning
the sentinel.  This proof check is intentional: the standalone reader can
continue after some failed forms, so a bare trailing sentinel would be a
false positive.  The default replay currently proves true =load= replay
for =lisp-mode.el=, =isearch.el=, =minibuffer.el=, and =project.el=.
Because the current standalone reader does not yet print Lisp errors,
the host-side runners treat reaching the sentinel exit code as success
and report crashes or non-sentinel exits as the failing input.  Set
=VENDOR_FORM_WALK_NORMALIZE_FLOATS=t= only for what-if diagnostics
that need to compare against integer-normalized forms; the default raw
path preserves the exact vendor form.  The previous raw
=simple.el= form 429 crash is fixed in the sibling standalone reader by
materializing reader floats as real =Sexp::Float= values.  Raw
standalone form-walk diagnostics now also reach the sentinel across the
full top-level form sets for =lisp-mode.el=, =isearch.el=,
=minibuffer.el=, and =project.el=; the true file-load replay for the
same default set is covered by =diagnose-vendor-load-replay=.
Supported batch/interactive runtime image bakes use
=NEMACS_RUNTIME_BAKE_TIMEOUT=900s=; source-v1 replay smokes use
=NEMACS_RUNTIME_REPLAY_TIMEOUT=900s= and longer interactive/vendor
variants.  Large pure-Elisp vendor loads run with
=NELISP_STACK_LIMIT=unlimited= by default; override that variable if a
platform rejects unlimited process stacks.
=bin/nemacs= resolves NeLisp in this order: explicit =NELISP_HOME= /
=NEMACS_NELISP=, =vendor/nelisp=, sibling =../nelisp=, then the legacy
=~/Notes/dev/nelisp= path.  Within a checkout it uses
=target/nelisp-standalone-reader=.

=make demo= / =make demo-phase2= prints the demo's observable state
plist so you can inspect the integration round-trip from the
shell — useful as a smoke test on a fresh clone.
