* Status — read this first

This is an *early, exploratory research project*.  It is *not* a
usable Emacs replacement and is unlikely to become one for several
years.  Most user-visible Emacs features are absent, the API is
unstable, and breaking changes are expected on most commits.

Concretely, as of 2026-05-14:

- Phase 1 (= core editor substrate: =buffer.c=, =window.c=, =frame.c=,
  =keymap.c=, =minibuffer.c= ported to elisp) is *in progress*.
- Phase 2 (= TUI reference display backend) has *scaffolding only*.
- Phases 3-5 (= ~xdisp.c~ port, MELPA shim, Emacs bootstrap /
  self-host) are *unstarted*.

If you are looking for a working pure-elisp Emacs binary today, this
is not it.  See [[#long-term-roadmap][Long-term roadmap]] below for an
honest timescale.

* What is this?

=nelisp-emacs= is the Layer 2 (extension package) + Layer 3 (TUI
reference display backend) of the
[[https://github.com/zawatton/nelisp][NeLisp]] stack.  The long-term
goal is to re-implement Emacs's core C code in elisp, running on the
NeLisp runtime, so that a pure-elisp Emacs binary can be produced
without a hand-written C core.

* Relationship to NeLisp

- =Layer 1 (NeLisp core runtime)= = upstream
  [[https://github.com/zawatton/nelisp][~nelisp~]] (vendored under
  =vendor/nelisp/= via =bin/build-nelisp=).  This repo never modifies
  Layer 1.
- =Layer 2 (extension package)= = partly upstream (the
  =nelisp-emacs-compat=, =nelisp-process=, =nelisp-regex=,
  =nelisp-coding=, =nelisp-sqlite=, =nelisp-base64=,
  =nelisp-secure-hash=, =nelisp-json=, =nelisp-filenotify=, …
  modules are part of the NeLisp distribution).  *This repo is the
  remaining Layer 2 work* — the Emacs interactive editor substrate
  (buffer / window / frame / keymap / minibuffer / etc.) ported
  from Emacs C.
- =Layer 3 (display backend)= = TUI primary (in this repo).  The
  GTK4 GUI sibling lives in
  [[https://github.com/zawatton/nelisp-gtk][~nelisp-gtk~]].  Cocoa /
  Win32 / =wgpu= backends are open to community contribution.

* Repository layout

- =src/= — Layer 2 / Layer 3 elisp implementation
  (=emacs-buffer.el=, =emacs-window.el=, =emacs-frame.el=,
  =emacs-keymap.el=, =emacs-minibuffer.el=,
  =emacs-tui-backend.el=, =emacs-tui-terminfo.el=,
  =emacs-network-ffi.el=, =emacs-sqlite-ffi.el=,
  =emacs-server-polyfills.el=, …)
- =test/= — ERT tests for the modules under =src/=
- =vendor/emacs-lisp/= — vendored copies of selected Emacs
  =lisp/= subtrees used as references during the port
  (e.g. =vendor/emacs-lisp/org/= for the day when an
  =org-mode= bootstrap becomes meaningful)
- =vendor/nelisp/= — produced by =bin/build-nelisp= (= =make nelisp=)
- =docs/design/= — design documents for each port phase
- =bench/= — micro-benchmarks (advisory; not merge gates)

The =vendor/= elisp is *imported, not implemented*.  Until the Layer
2/3 substrate is functional enough to host those packages, the vendored
copies are reference material — they will not produce a working
=org-mode= or =Org agenda= on top of =nelisp-emacs= today.

* Initial scope (Phase 1)

Per =docs/design/01-architecture.org= (LOCKED):

1. =src/emacs-buffer.el= = =buffer.c= port (buffer-local
   variables, advanced text-property, full undo)
2. =src/emacs-window.el= = =window.c= port
3. =src/emacs-frame.el= = =frame.c= port (display backend
   abstraction)
4. =src/emacs-keymap.el= = =keymap.c= port
5. =src/emacs-minibuffer.el= = =minibuffer.c= port
6. =src/emacs-tui-backend.el= = TUI display backend MVP
   (ncurses-style, reference impl)

* Long-term roadmap
:PROPERTIES:
:CUSTOM_ID: long-term-roadmap
:END:

These estimates are *aspirational*, not commitments.  Each phase
boundary is a real piece of work — the figures below assume single
maintainer evening hours.

| Phase   | Scope                              | LOC estimate | Wall-clock |
|---------+------------------------------------+--------------+------------|
| Phase 1 | Layer 2 core (buffer / window / …) | ~5-10K       | 6-12 mo    |
| Phase 2 | TUI backend reference impl         | ~5-10K       | 6-12 mo    |
| Phase 3 | =xdisp.c= port (display engine)    | ~10-30K      | 12-24 mo   |
| Phase 4 | MELPA compat layer (shim)          | ~20-50K      | 1-2 yr     |
| Phase 5 | Emacs bootstrap (self-host)        | full         | 3-5 yr     |

In particular: =org-mode= depends on Phase 1 (buffer + text
properties) + Phase 2 (TUI redisplay) + Phase 3 (display engine, for
visual folding / agenda / table alignment) all reaching usable
state.  Today, ~ert-run~ batch-style use of vendored elisp is
plausible; interactive editing in =nelisp-emacs= is not.

* Building

#+begin_src sh
make nelisp        # populate vendor/nelisp/ from upstream
make compile       # byte-compile src/
make test          # run the ERT test suite
#+end_src

The nelisp binary tool is the source of truth for
=vendor/nelisp/=; do not add Makefile targets or scripts that assume
an external nelisp checkout outside =$NEMACS_HOME=.
